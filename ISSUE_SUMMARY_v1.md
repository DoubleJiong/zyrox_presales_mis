## 1. 系统业务理解

从文档和代码综合判断，该系统试图覆盖售前业务的主链路：

1. 线索 -> 商机 -> 项目
2. 项目从商机阶段进入投标、执行、结算、归档
3. 解决方案既有早期 `scheme` 体系，也有后期 `solution` 体系
4. 投标、报价、方案评审、人员协作、绩效与仲裁属于项目周边管理能力
5. 系统同时承担前端页面、后端 API、认证、权限、数据库访问和部分运维能力

整体上，这不是一个简单的表单系统，而是一个“业务流 + 权限流 + 审批流 + 数据统计”混合型单体应用。

## 2. 问题分级说明

- P0：高危问题，可能导致越权、数据泄露、核心业务失真或生产事故
- P1：重要问题，影响关键业务流程、可维护性或统计口径
- P2：中等问题，短期不一定阻塞上线，但会持续积累风险

## 4. 重点业务问题展开

### 4.1 项目阶段流转不严谨

项目表已经引入 `projectStage`，希望表达“商机 -> 投标 -> 执行 -> 结算 -> 归档”的业务链路，但相关 API 并没有形成一个严格状态机。

当前问题：

1. 投标立项审批在提交时直接把阶段改为 `bidding`
2. 审批通过时只是补一条投标记录，并没有完整校验前置条件
3. 审批拒绝时又直接改回 `opportunity`
4. 与 `status` 字段之间没有稳定同步机制

这会导致“阶段看起来已经进入投标，但审批过程并没有沉淀为可追踪的业务记录”。

### 4.2 无负责人项目的可见性规则有业务风险

项目列表对普通用户开放了“无负责人项目”，逻辑上像是在方便认领，但在售前场景里，无负责人项目往往恰恰是最敏感的早期商机。

这类数据通常应当：

1. 仅管理员或指定角色可见
2. 必须通过认领流程变更负责人
3. 保留认领前后的操作记录

当前实现会导致敏感项目在组织内被超范围浏览。

### 4.3 方案体系分裂

系统中既有 `scheme`，也有 `solution`。从数据库命名、服务命名和 API 命名看，两套体系并未完全清理或完成迁移。

这会带来三类业务风险：

1. 用户口径不一致：前台叫“解决方案”，后台表可能仍是“方案”
2. 统计口径混乱：仪表盘、排行、审批可能统计不同对象
3. 生命周期错乱：旧对象与新对象可能并存，却没有主从关系或迁移策略

### 4.4 人员与账号模型偏“演示系统”而非正式业务系统

当前人员创建允许默认密码 `123456`，登录页还预填并展示测试账号密码。这类实现说明系统里仍残留较多 demo 假设。

对正式业务系统来说，人员管理至少应满足：

1. 创建后强制首次改密
2. 管理员不能看到明文默认口令
3. 测试账号不进入正式 UI
4. 角色、岗位、组织、业务职责的关系清晰可审计

## 6. 当前审核重点

建议人工审核重点集中在以下四项：

1. 真实角色环境权限复核：确认自动化验证之外，关键角色看数、看项目、审批处理权限符合预期。

3. 迁移证据复核：确认 `001` 到 `007` 已全部登记为 `applied`，并核对 legacy 定向备份产物。
4. 发布观察项确认：明确正式发布时间、值守安排和观察窗口责任人。
