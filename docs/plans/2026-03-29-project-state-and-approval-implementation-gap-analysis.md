# 项目状态机与审批体系实现偏差分析

**目标：** 将第二阶段的状态机/审批设计与当前代码实现之间的偏差明确化，为第三阶段重构切片提供直接输入。

---

## 1. 目标设计来源

本分析以 `2026-03-29-project-state-machine-and-approval-system-design.md` 为唯一设计来源。

目标设计要求：

1. `projectStage` 为唯一业务主阶段字段
2. 审批与状态推进解耦
3. 审批有主表、步骤表、事件表
4. 阶段变化必须可审计、可测试、可回放

## 2. 当前主要偏差

### 偏差 1：项目存在双主语义字段

代码现状：

1. `src/db/schema.ts` 中 `projects.projectStage` 已存在
2. 同一表中 `projects.status` 仍被广泛读取和聚合
3. `src/services/query-service.ts` 和其他列表/统计逻辑仍继续消费 `status`

影响：

1. 统计口径分裂
2. 权限判断可能引用不同字段
3. 业务流转难以形成单一真相源

第二阶段结论：

1. `projectStage` 为唯一主阶段字段
2. `status` 退化为兼容字段

### 偏差 2：投标审批并未形成审批实体

代码现状：

1. `src/app/api/biddings/approvals/route.ts` 通过直接修改项目字段来表达“提交审批”
2. 审批列表 ID 使用前端拼装式 `index + 1`
3. 项目进入投标审批时，直接把 `projectStage` 改到 `bidding`

影响：

1. 审批和阶段变化混为一体
2. 无法形成真实审批审计链
3. 无法可靠支持撤回、重提、超时、流转历史

第二阶段结论：

1. 投标立项审批必须迁移到独立 `approval_request` 体系
2. `bidding_pending` 与 `bidding` 必须区分
3. 审批通过才允许从 `bidding_pending` 进入 `bidding`

### 偏差 3：方案评审仍是局部状态回写

代码现状：

1. `src/services/solution-review.service.ts` 已基于 `solutionReviews` 工作
2. 但评审完成后，仍只按 `review.isFinal` 局部回写 `solutions.status`
3. 方案评审未纳入统一审批实体和统一事件模型

影响：

1. 评审链路无法与项目阶段统一
2. 无法将评审待办作为正式审批待办对待
3. 无法稳定承接撤回、重提、多轮审批与审计需求

第二阶段结论：

1. `solution_review` 必须纳入统一审批框架
2. 方案状态变化必须有事件来源，而不是单纯状态回写

### 偏差 4：阶段历史与审批历史未成为统一事件源

代码现状：

1. schema 中已有 `projectStageHistory` 等历史结构
2. 但实际流程入口并未统一走状态机服务
3. route 直接更新字段的做法仍然存在

影响：

1. 历史记录不一定完整
2. 通知、统计、审计无法共享单一事件来源

第二阶段结论：

1. 第三阶段必须建立统一状态机服务
2. 所有阶段迁移统一通过状态机服务写入项目主表和阶段历史

## 3. 第三阶段第一批替换切片

### Slice 1：项目状态机服务

目标：停止 route 直接改项目阶段。

范围：

1. 建立 `project-stage-service`
2. 封装允许迁移关系、前置条件和历史写入
3. 让 `projectStage` 成为唯一阶段真相源

### Slice 2：统一审批主实体

目标：建立投标立项、方案评审、合同审批共享的审批骨架。

范围：

1. 新建 `approval_request` / `approval_step` / `approval_event`
2. 将投标立项从 `api/biddings/approvals` 迁移为审批单驱动
3. 保留旧接口兼容层，但不再直改项目阶段

### Slice 3：方案评审接入统一审批

目标：让方案评审从“局部服务”升级为“统一审批类型”。

范围：

1. 复用统一审批骨架
2. 保留 `solution_review` 业务语义
3. 让评审结果驱动状态与事件，而不是仅回写状态字段

### Slice 4：统计与仪表盘切换到新口径

目标：让 dashboard 消费状态机与审批真相源。

范围：

1. 项目统计改为基于 `projectStage`
2. 审批待办改为基于审批主表
3. 旧 `status` 统计仅保留兼容展示

## 4. 第二阶段结论

第二阶段已经明确：当前问题不是“某个接口写错了”，而是项目阶段、审批实体、方案评审和统计入口尚未收敛到单一模型。第三阶段必须按“状态机服务 -> 统一审批实体 -> 方案评审接入 -> 仪表盘切换”顺序推进，禁止再对现有 route 做无边界补丁式扩展。
