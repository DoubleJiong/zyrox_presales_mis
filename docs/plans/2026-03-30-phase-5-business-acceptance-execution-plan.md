# 第五阶段业务闭环验收执行方案

日期：2026-03-30

## 1. 目标

本轮执行不再停留在“接口能返回”和“页面能打开”。

验收目标改为：

1. 验证关键页面在真实业务账号下的前后台数据一致性。
2. 验证页面操作后数据写入、聚合计算、权限过滤、刷新重载是否与业务预期一致。
3. 验证统计页面与明细页面是否使用同一业务口径，而不是各算各的。
4. 形成可复验证据，而不是凭主观观感判断“看起来差不多”。

## 2. 执行原则

1. 不做全系统无差别漫游式测试，按主链路和统计口径分层推进。
2. 每个结论必须至少有 4 层证据中的 3 层支撑：页面表现、接口响应、数据库事实、人工台账事实。
3. 页面问题与口径问题分开记录，不允许把展示错误误判成业务数据错误。
4. 所有校验都以业务账号视角为主，管理员仅用于对照和排障。

## 3. 执行分层

### 第一层：对象主链一致性

对象范围：客户、项目、方案。

目标：确认主对象在页面、接口、数据库之间是一致的，且刷新后仍保持一致。

每个对象按以下 6 个检查点执行：

1. 首次加载：页面首屏数量、默认排序、默认过滤条件是否与接口一致。
2. 条件筛选：搜索、状态筛选、分页切换后，页面展示条数是否与接口分页和数据库事实一致。
3. 详情跳转：列表项进入详情后，主字段是否与列表一致。
4. 编辑/新增：一次关键字段修改后，列表、详情、关联对象是否同时反映。
5. 刷新重载：浏览器刷新或重新登录后，数据是否持久存在且显示不回退。
6. 权限视角：业务账号与管理员对同一对象的差异，是否仅来自范围差异，而不是公式差异。

### 第二层：跨页面联动一致性

对象链：客户 -> 项目 -> 方案。

目标：确认同一业务样本在不同页面中引用的是同一事实。

检查点：

1. 项目列表中的客户名称，能在客户列表中找到对应记录。
2. 方案列表中的关联项目，能在项目列表和项目详情中找到同一项目。
3. 项目详情中的方案信息与方案列表中的主字段一致。
4. 任何一处修改后，其他页面在刷新后能够反映最新值。

### 第三层：统计口径一致性

对象范围：仪表盘、统计卡片、分布图、列表总数。

目标：确认统计数字来自统一口径，而不是页面各自拼装。

本轮至少校验以下统计项：

1. `totalCustomers`
2. `totalProjects`
3. `projectsByStage`
4. `totalSolutions`
5. 列表页总数与分页总数
6. 业务样本抽检中的阶段分布与归档结果

每个统计项按以下顺序校验：

1. 读口径定义文档。
2. 确认接口字段来源。
3. 直接查询数据库对象集合。
4. 与页面展示值对比。
5. 对不一致项判断属于：权限范围差异、展示错误、聚合错误、样本导入错误，还是业务口径未定义。

## 4. 样本设计

本轮不采用随机抽样，采用“代表性样本集”。

样本至少覆盖：

1. 商机阶段项目
2. 招投标阶段项目
3. 归档阶段项目
4. 中标样本
5. 丢标样本
6. 放弃样本
7. 有项目方案的样本
8. 无项目方案的样本
9. 不同客户类型样本
10. 不同项目类型样本

建议本轮先形成 15 到 20 条样本清单，作为业务人工核对主样本。

## 5. 单条样本的校验模板

每条样本按固定模板记录：

1. 台账标识：项目名称、客户名称、原台账状态
2. 系统对象：客户 ID、项目 ID、方案 ID
3. 页面证据：客户页、项目页、方案页是否可见
4. 接口证据：对应 API 返回的关键字段
5. 数据库证据：主表记录和关联表记录
6. 统计证据：该样本是否计入相关统计卡片和阶段分布
7. 判定：一致 / 不一致
8. 根因分类：导入映射、权限过滤、接口聚合、前端展示、口径定义缺失

## 6. 页面校验方法

### 6.1 明细页校验

对客户页、项目页、方案页，执行以下动作：

1. 打开页面，记录初始总数和首屏数据。
2. 输入筛选条件，验证页面结果与接口返回结果一致。
3. 打开任一代表性样本详情，记录关键字段。
4. 对可编辑字段做一次最小变更。
5. 返回列表页并刷新，确认修改结果没有只停留在前端本地状态。

### 6.2 统计页校验

对仪表盘、统计卡片、阶段图，执行以下动作：

1. 记录页面展示值。
2. 调对应 API。
3. 用数据库查询复算一次。
4. 对照同口径列表页总数。
5. 记录偏差来源和影响范围。

## 7. 具体开展顺序

执行顺序必须固定，不建议跳步：

1. 先锁定样本集。
2. 再做客户/项目/方案明细链核对。
3. 再做跨页面联动核对。
4. 再做统计卡片和阶段分布核对。
5. 最后做一次修改后刷新复验，验证不是前端缓存假象。

原因：

1. 若主对象事实还未核准，先看仪表盘没有意义。
2. 若跨页面引用不一致，统计页即使数字对，也不能算业务闭环通过。

## 8. 本轮我方执行方式

我会按以下方式推进：

1. 先从当前台账导入数据中挑出代表性样本清单。
2. 为每条样本准备“台账事实 / 系统事实 / 页面入口”三联表。
3. 用业务账号逐页核对客户、项目、方案主链。
4. 对每个统计项给出数据库复算口径，避免只信页面数字。
5. 把发现的问题按以下类别拆开：
   - 页面渲染问题
   - 接口契约问题
   - 数据权限问题
   - 台账导入映射问题
   - 统计聚合问题
6. 每收口一批问题后立即回归复验，不留到最后统一复测。

## 9. 产出物

本轮应形成以下交付物：

1. 代表性样本清单
2. 客户/项目/方案主链核对记录
3. 统计口径复算记录
4. 不一致问题台账
5. Phase 5 最终业务闭环结论

## 10. 阶段判断标准

只有同时满足以下条件，才能认为这一步完成：

1. 代表性样本在客户/项目/方案三条链路上可追溯。
2. 页面数值、接口数值、数据库复算数值之间无未解释差异。
3. 已知差异项均已定性，并明确是否阻断 Phase 5。
4. 业务侧可以基于样本记录作出“通过 / 不通过”判断，而不是继续补猜测。
