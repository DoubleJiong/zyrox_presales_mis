# 售前管理系统后续路线评估：重启新项目，还是在原系统上迭代升级

**目标：** 基于现有评估与改造方案，判断该系统下一步更适合“新开一个新版本重建”，还是“在原系统上分阶段迭代升级”。

---

## 1. 结论先行

我的结论是：

**不建议直接推倒重来，也不建议继续在当前系统上无约束修补。最合理的路线是：以现有系统为基底，采用“受控重构 + 分阶段替换”的升级策略。**

换句话说：

1. 不是原地小修小补。
2. 也不是立刻开一个完全平行的新项目重写。
3. 而是以当前系统为业务壳，先止血、再定规则、再逐步替换核心域。

这是我目前最明确的建议。

## 2. 为什么不建议直接重启一个全新项目

如果现在立刻重启一个新项目，表面上看会更干净，但实际风险并不低。

### 2.1 业务规则还没有完全定版

当前最大问题不是技术栈过时，而是业务规则本身还没有被彻底澄清。

如果在此时重写：

1. 很容易把当前系统中的混乱规则原样复制到新系统。
2. 甚至会把一些尚未暴露的问题带进新架构。
3. 新系统会获得更好的代码外观，但不一定获得更正确的业务内核。

### 2.2 当前系统已经沉淀了一部分真实业务线索

即使它还不成熟，当前系统仍然已经暴露了很多真实问题：

1. 哪些链路是常用的
2. 哪些权限点有风险
3. 哪些统计口径容易失真
4. 哪些状态设计不合理

这些信息非常有价值。如果现在完全另起炉灶，团队很容易在新项目中重新踩一遍相同的坑。

### 2.3 完全重写的组织成本更高

重启一个新项目意味着：

1. 新架构设计
2. 新数据迁移方案
3. 新测试体系
4. 双系统并存期
5. 用户迁移与培训
6. 历史数据兼容

对于当前这个系统来说，真正贵的往往不是写代码，而是重新建立业务信任和迁移秩序。

## 3. 为什么也不建议直接在原系统上继续边修边加

另一种常见误区是：既然不重写，那就继续在原系统上修一点、补一点、加一点。

这同样不合理。

### 3.1 当前系统已经到达“再无序增量会迅速恶化”的拐点

它当前的问题不是模块少，而是规则和边界没有收敛。如果继续用旧方式堆需求：

1. 双实现会继续变多
2. 统计口径会继续分裂
3. 权限例外会越来越多
4. 历史兼容逻辑会越来越厚

### 3.2 AI 生成系统最怕“局部修补替代系统治理”

AI 生成的系统往往局部很完整，因此团队容易产生一种错觉：哪里坏了就修哪里。

但这类系统真正的问题常常在于：

1. 全局模型没有统一
2. 共享能力没有统一
3. 状态与审批不是一套规则

如果只修局部，系统会越来越像补丁集合。

## 4. 推荐路线：受控重构 + 分阶段替换

### 4.1 路线定义

建议采用“在原系统上演进，但按新规则重建核心域”的方式推进。

具体含义是：

1. 保留现有仓库和现有业务壳，不立即另起一个完全平行的新项目。
2. 先完成第一阶段止血，恢复系统可控性。
3. 再以新定义的状态机、审批体系、权限矩阵为准，重构核心业务域。
4. 对最混乱、最关键的模块采用“新模块替换旧模块”的方式演进。
5. 在必要时允许形成“旧逻辑兼容层 + 新逻辑主实现”并行一段时间。

### 4.2 哪些部分适合原地升级

以下部分建议原地升级：

1. 基础工程体系：CI、测试、配置、日志、监控
2. 认证与会话体系
3. 通用校验与错误处理
4. 文档、脚本、部署与运行手册
5. 页面壳与导航结构

这些部分更像基础设施，适合直接在原系统中收敛。

### 4.3 哪些部分适合“局部重建式升级”

以下部分不建议只做小修，建议按新模型重建后替换：

1. 项目主链路状态管理
2. 投标立项审批
3. 方案评审体系
4. 权限判定核心逻辑
5. 仪表盘关键统计口径

这些模块已经不仅仅是代码脏，而是模型本身可疑，适合按新设计重做。

## 5. 什么时候才建议新开一个完全新版本项目

只有在以下条件同时成立时，我才会建议完全新开项目：

1. 现有代码无法稳定构建和运行，且短期无法止血。
2. 现有数据模型与目标业务模型冲突过大，几乎无法迁移。
3. 团队准备接受双系统长期并行和完整迁移成本。
4. 业务规则已经高度稳定，足以支撑一次性重建。
5. 现有系统用户量和历史数据量还不大，切换成本可控。

以目前掌握的信息看，这五个条件并没有同时满足。

## 6. 当前更推荐的判断模型

我会把你的选项从“重写 or 不重写”改成下面三档：

### 选项 A：原地继续修补

不推荐。

原因：风险最低看似最低，实则会把问题继续滚大。

### 选项 B：受控升级重构

强烈推荐。

原因：

1. 保留已有业务认知和系统外壳
2. 控制迁移成本
3. 允许边治理边验证
4. 可逐步替换最有问题的核心域

### 选项 C：全新项目重建

暂不推荐作为当前第一选择。

原因：当前业务规则未完全定版，贸然重建容易把错误更系统化地复制一遍。

## 7. 推荐实施策略

### 阶段 A：先止血

目标：让旧系统可控。

动作：

1. 安全止血
2. 认证统一
3. 构建修复
4. 测试起盘

### 阶段 B：再定版

目标：定义正确规则。

动作：

1. 项目状态机定版
2. 审批体系定版
3. 权限矩阵定版
4. 统计口径定版

### 阶段 C：再替换核心域

目标：把旧逻辑逐步替换成新逻辑。

动作：

1. 新项目流转模块上线
2. 新审批模块上线
3. 新权限策略上线
4. 新统计逻辑上线

### 阶段 D：最后再评估是否需要版本化重构

当上述三阶段做完后，再评估是否需要：

1. 开 v2 仓库
2. 拆服务
3. 大规模前端重构

那个时候做判断会更准确，也更便宜。

## 8. 判断依据汇总

我之所以做出“优先受控升级，不优先完全重写”的判断，基于以下事实：

1. 当前系统能跑起部分主链路，说明它不是纯空壳。
2. 当前最大问题在规则和边界，不在技术栈本身。
3. 现有问题仍然足以通过阶段治理逐步收敛。
4. 现在立即重写，会在业务未定版的情况下放大不确定性。
5. 当前更需要的是治理能力，而不是第二个代码仓库。

## 9. 管理层视角下的建议结论

如果用更管理化的话来表达，建议是：

**把当前系统视为“v1 业务实验系统”，不要把它继续当成“最终成品”，也不要马上抛弃它重写成 v2。更合理的做法是：在 v1 上完成治理升级，并通过受控替换逐步长成真正可生产的版本。**

这意味着：

1. 当前仓库继续保留
2. 但后续每个核心域都按“准 v2 标准”重构
3. 当核心域替换到一定程度后，再决定是否正式命名为新版本

## 10. 最终建议

基于当前评估与已形成的实施计划，我的最终建议是：

**下一步优先选择“在原有系统上迭代升级，但采用新版本治理标准”，而不是直接启动一个完全平行的新项目。**

更直白一点：

1. 仓库先不换
2. 方法必须换
3. 规则先定
4. 核心域重做
5. 基础设施先补齐
6. 等系统收敛后，再决定是否名义上切到 v2

这比现在直接重启一个新项目，更稳、更省、更符合当前真实情况。
