# 售前管理系统生产化改进方案

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 将当前由 AI 快速生成、已具备业务外形但尚不稳定的售前管理系统，收敛为一个业务规则清晰、工程边界稳定、安全可控、可持续交付的真实生产环境系统。

**Architecture:** 当前系统不适合直接在现状上持续堆功能，必须先完成“业务真相澄清 + 模型收敛 + 安全治理 + 工程硬化”四条主线的联合治理。整体策略建议采用“单体内重构优先、边界先收敛、能力后扩展”的路线，在现有 Next.js 单体中完成业务对象、鉴权、状态机、测试与发布链路的系统整理，避免过早拆分为微服务而增加复杂度。

**Tech Stack:** Next.js 16、React 19、TypeScript 5、Node.js、Drizzle ORM、PostgreSQL、Playwright、Vitest、Docker、CI/CD

---

## 1. 方案定位

这不是一份“修 bug 清单”，而是一份“接盘 AI 生成系统并把它带到生产级”的系统方案。

基于当前审查结果，我对系统的定位如下：

1. 它已经具备真实业务系统的外形和部分可运行能力。
2. 它尚未具备真实生产系统所要求的稳定边界、统一模型和交付可信度。
3. 它当前最主要的问题不是某个单点缺陷，而是业务规则、权限规则、状态规则和工程规则没有收敛成统一系统。
4. 因此，后续工作不能只按“修功能”推进，而必须按“业务重构 + 架构收敛 + 工程治理 + 上线验收”推进。

## 2. 总体目标

本方案的最终目标不是“让系统能跑”，而是让系统满足以下五个标准：

1. 业务上：核心对象、状态、审批和统计口径有统一定义。
2. 安全上：认证、授权、配置、审计、测试后门和敏感数据处理满足生产要求。
3. 工程上：可以稳定构建、稳定测试、稳定发布，并支持回归验证。
4. 运行上：关键链路有监控、有日志、有告警、有回滚能力。
5. 组织上：系统行为不再依赖少数人口头解释，而能由文档、代码和测试共同约束。

## 3. 当前系统成熟度判断

基于已有调研，当前系统更接近以下阶段：

- 不是概念验证系统。
- 不是成熟生产系统。
- 更接近“高仿真业务原型”或“试运行级内部系统”。

其典型特征包括：

1. 模块覆盖广，但全局一致性不足。
2. 局部能力可运行，但跨对象规则不稳定。
3. 文档、代码、脚本、测试资产存在漂移。
4. 安全、运维、测试边界存在明显演示化痕迹。
5. 一部分业务逻辑更像 AI 推理出的“合理形状”，未必是组织真实规则。

## 4. 改造总原则

### 4.1 先收敛，再扩展

在业务主链路、权限模型、状态机、统计口径未统一前，原则上不建议继续扩展大量新功能。

### 4.2 先保单体内边界，再考虑拆分

当前最合理的路线不是拆微服务，而是在现有单体中完成业务模块边界整理、技术能力统一和测试补齐。过早拆分只会把不清晰的规则复制到更多服务中。

### 4.3 以业务规则为第一约束

所有技术重构都必须服务于业务规则澄清，而不是反过来让业务去适配现有代码结构。

### 4.4 以“可验证”为准，不以“感觉更好”为准

每一轮改造都必须有明确验收物：流程图、数据字典、权限矩阵、自动化测试、发布门槛、性能与错误率指标。

## 5. 建议采用的目标架构

### 5.1 目标形态

建议未来 6 到 12 周内，把系统收敛到如下形态：

1. 前端：保留 Next.js App Router，但按业务域重组页面与状态流。
2. 后端：保留 Next.js Route Handlers 作为单体 API 层，但统一中间件、鉴权、校验、异常处理与审计。
3. 领域层：形成清晰的 domain/service/repository 边界，避免业务逻辑散落在 route 中。
4. 数据层：统一数据字典、主数据关系、状态机定义、审计字段和迁移策略。
5. 运维层：补足环境配置、构建、容器化、发布、监控、告警与回滚机制。
6. 测试层：形成单元、集成、接口、E2E 四层测试结构。

### 5.2 高层结构

```text
UI / App Router
    |
Application Layer (Route Handlers / Controllers)
    |
Domain Services
    |
Repositories / Drizzle ORM
    |
PostgreSQL

Cross-Cutting Concerns:
- AuthN / AuthZ
- Validation
- Audit Log
- Error Handling
- Observability
- Feature Flags
- Test Harness
```

### 5.3 架构决策

#### ADR-01：短期内继续使用模块化单体

- 决策：不拆微服务，先做模块化单体治理。
- 原因：当前核心问题是模型不清、规则不稳，不是服务拆分不足。
- 收益：降低迁移成本，便于快速收敛业务边界。
- 风险：如果单体治理不到位，代码仍可能继续膨胀。
- 约束：必须引入模块边界与目录规范，否则单体只会继续恶化。

#### ADR-02：认证与授权统一为一套实现

- 决策：删除双 JWT 体系和 query token 方案，统一令牌生成、校验、刷新、注销、审计。
- 原因：当前认证链路分裂，边界不可控。
- 收益：降低安全风险，提高排障和测试可信度。
- 风险：会触及较多接口和前端会话逻辑。

#### ADR-03：项目流程改为显式状态机

- 决策：把项目阶段、状态、审批关系建模为明确状态机，而不是多个字段的松散组合。
- 原因：当前最核心的业务失真来自状态推进规则不清。
- 收益：为权限、统计、审批、通知提供统一依据。
- 风险：需要补数据迁移和历史数据矫正方案。

#### ADR-04：方案体系收敛为单一主模型

- 决策：明确 `scheme` 与 `solution` 的保留/废弃关系，最终保留单一主模型。
- 原因：双模型并存会长期破坏统计、流程和认知一致性。
- 收益：降低维护和扩展成本。
- 风险：涉及历史接口兼容和数据迁移。

## 6. 改造范围与工作流

建议把改造工作拆为六条并行但有依赖关系的工作流。

### 工作流 A：业务真相澄清与模型重构

目标：搞清楚系统到底要表达什么业务，而不是沿用现有代码假设。

关键输出：

1. 核心业务对象清单
2. 术语表和统一命名
3. 流程图与状态机
4. 权限矩阵
5. 统计口径说明
6. 数据字典

核心对象建议先覆盖：

1. 客户
2. 线索
3. 商机
4. 项目
5. 投标立项/审批
6. 解决方案
7. 报价/合同
8. 人员与角色
9. 任务/协作
10. 绩效/仲裁

必须先回答的业务问题：

1. 项目与商机是什么关系，谁驱动谁。
2. `status` 和 `projectStage` 哪个是主状态，另一个是否保留。
3. 投标审批是“留痕实体”还是“阶段推进器”。
4. 无负责人项目是否可见，若可见，谁可见、如何认领。
5. `scheme` 和 `solution` 是否代表同一业务对象的不同历史阶段。
6. 仪表盘中的每个指标由什么规则计算。

### 工作流 B：安全治理与身份边界重建

目标：把系统从“演示化安全”拉回到“生产级安全”。

必须完成的事项：

1. 去除硬编码数据库连接串。
2. 下线或严格封禁测试后门接口。
3. 合并认证实现，统一 JWT 策略。
4. 禁止 URL query token。
5. 重建注销、黑名单与会话失效逻辑。
6. 引入密码策略、首次改密与账号生命周期管理。
7. 增加操作审计日志。
8. 梳理 RBAC 或 RBAC + ABAC 规则。
9. 为敏感接口增加权限和输入验证测试。

建议安全基线：

1. 所有环境变量分环境管理。
2. 所有敏感操作都有审计。
3. 所有管理接口默认最小权限。
4. 所有鉴权失败返回统一错误格式。
5. 所有会话变更可追踪。

### 工作流 C：领域收敛与代码结构重构

目标：减少“页面/路由直接承载业务规则”的现象，形成可维护的系统结构。

建议目录方向：

```text
src/
  app/
  modules/
    project/
    bidding/
    solution/
    customer/
    staff/
    dashboard/
  shared/
    auth/
    validation/
    audit/
    errors/
    observability/
  db/
```

每个业务模块至少包含：

1. domain types
2. service
3. repository
4. validator
5. policy
6. tests

重构重点：

1. route 中只保留协议转换与调用编排。
2. 核心业务逻辑迁移到 service。
3. 数据访问统一到 repository。
4. 通用错误处理、鉴权、日志和校验统一放入 shared。

### 工作流 D：数据治理与迁移治理

目标：让数据库从“能存数据”变成“能支撑真实业务”。

必须产出的内容：

1. 逻辑数据模型
2. 物理表结构整改方案
3. 历史数据清洗方案
4. 迁移脚本策略
5. 回滚策略
6. 审计字段和软删除策略

高优先级数据问题：

1. `customerId` 与 `customerName` 双存储的治理策略
2. `scheme` 与 `solution` 的迁移路线
3. 项目状态与审批记录的关系重建
4. 默认测试账号和历史测试数据清理
5. 统计口径依赖的历史数据修正

### 工作流 E：工程硬化与交付体系建设

目标：让系统具备持续开发、持续验证和持续发布能力。

必须完成的事项：

1. 修复生产构建。
2. 修复开发态 Turbopack 稳定性问题。
3. 补齐 `.env.example`、启动脚本、部署脚本和运维文档。
4. 增加 lint、typecheck、unit、integration、e2e 到 CI。
5. 容器化本地环境与测试环境。
6. 建立分支策略、代码评审规则和发布门槛。
7. 增加错误追踪、结构化日志和核心指标监控。
8. 建立数据库备份与恢复演练。

建议发布门槛：

1. `build` 必须通过。
2. 核心冒烟测试必须通过。
3. 关键权限测试必须通过。
4. 数据迁移有演练记录。
5. 可以在 30 分钟内完成回滚。

### 工作流 F：测试体系与验收体系建设

目标：把“感觉可用”变成“可验证可回归”。

建议测试分层：

1. 单元测试：状态机、鉴权策略、校验规则、统计口径函数
2. 集成测试：service + repository + database
3. API 测试：登录、项目、审批、方案、人员、合同等关键接口
4. E2E 测试：登录、项目创建、立项审批、方案评审、仪表盘查看
5. 生产前验收：真实业务样例回放和权限回归

最低测试要求：

1. 核心主链路必须有自动化回归
2. 每个高危权限点必须有负路径测试
3. 每个关键统计指标必须有样例校验
4. 每次发布必须执行冒烟套件

## 8. 关键业务重构建议

### 8.1 项目主链路重构

建议把“线索 -> 商机 -> 项目 -> 投标 -> 执行 -> 结算 -> 归档”定义为明确主链路，并要求：

1. 每个阶段只有明确进入条件和退出条件。
2. 每次阶段变化必须有事件记录。
3. 审批不直接替代阶段，而是成为阶段变化的条件之一。
4. 统计统一基于同一主链路状态。

### 8.2 审批体系重构

当前审批更像散落在业务中的局部动作。建议把审批变成真正的一等实体：

1. 有审批单主表
2. 有审批步骤
3. 有审批结果
4. 有驳回、撤回、重提、超时等状态
5. 有审批审计日志

### 8.3 方案体系重构

建议明确以下二选一结论：

1. `scheme` 只是历史遗留命名，应迁移到 `solution`。
2. `scheme` 与 `solution` 是不同业务对象，但必须明确边界、上下游和统计口径。

如果做不到这点，后续所有仪表盘、排行、评审和权限逻辑都会持续失真。

### 8.4 人员与权限体系重构

建议把用户、员工、角色、岗位、部门、业务职责这几类对象拆清楚，至少做到：

1. 谁能登录
2. 谁能看什么
3. 谁能审批什么
4. 谁对项目负责
5. 谁只参与执行

当前系统在这方面仍偏“账号系统”，还没有真正成为“组织权限系统”。

## 9. 生产级验收标准

只有当以下条件同时满足，系统才建议视为“可进入稳定生产环境”：

### 9.1 业务验收

1. 核心流程文档已定版
2. 状态机与权限矩阵已定版
3. 仪表盘关键指标有对账样本
4. 试运行用户确认流程可用

### 9.2 工程验收

1. `lint`、`typecheck`、`build`、测试全部通过
2. 核心 E2E 冒烟稳定通过
3. 发布流程自动化
4. 关键错误有监控和告警

### 9.3 安全验收

1. 无默认口令
2. 无公开测试后门
3. 无硬编码敏感配置
4. 敏感操作全量审计
5. 权限回归测试通过

### 9.4 运维验收

1. 有备份和恢复演练
2. 有回滚预案
3. 有运行手册
4. 有 on-call 与故障处理机制

## 10. 组织与协作建议

如果要真正把这个系统带到生产级，建议至少形成以下角色协同：

1. 业务负责人：确认业务真相和口径
2. 产品负责人：收敛需求和流程
3. 技术负责人：控制架构、边界和节奏
4. 开发负责人：执行模块改造
5. 测试负责人：建立验收样例和回归体系
6. 运维负责人：建立发布、监控、回滚机制

没有这些角色分工，系统容易再次回到“谁发现问题谁临时改一点”的状态。

## 11. 风险与应对

### 风险 1：业务方无法快速统一规则

应对：先冻结争议最大的几个对象，优先定义主链路、权限和统计口径，不要求一次性覆盖所有细枝末节。

### 风险 2：历史数据质量太差，迁移困难

应对：先建立数据分层策略，允许“历史兼容区 + 新模型区”并行一段时间，但必须有明确退场计划。

### 风险 3：团队继续边改边加新需求

应对：设立变更闸门，未通过架构和业务评审的新需求不进入核心链路。

### 风险 4：AI 代码太多，局部修补成本越来越高

应对：优先改造业务核心域，而不是平均修所有模块；先抓项目、审批、方案、权限四个关键域。

## 12. 第一批必须立刻启动的事项

以下事项建议在下一个迭代立即启动：

1. 召开业务规则澄清会，确认项目、审批、方案、权限四个核心对象。
2. 建立系统术语表、状态机草案和权限矩阵草案。
3. 下线测试后门，去掉硬编码配置，统一认证方案。
4. 修复生产构建和登录页浏览器渲染稳定性问题。
5. 建立最小 CI：`lint`、`typecheck`、`build`、关键冒烟测试。
6. 选定一条真实业务链路做试点回归。

1. 项目状态机设计文档

## 14. 结论

如果把当前系统视为“AI 生成的业务原型”，那最正确的接盘方式不是继续堆代码，而是先把它收敛成一个真正被定义过、验证过、约束过的系统。

这项工作是可以做成的，但前提不是“多修几个 bug”，而是接受一个事实：

当前系统的核心任务已经不是开发功能，而是完成一次从原型到产品级系统的治理升级。

只要业务规则能被澄清、技术边界能被统一、测试和发布体系能被建立，这个系统仍然有机会被带到稳定可靠的真实生产环境。
