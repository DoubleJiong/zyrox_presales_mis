# 工作台、日程、任务、消息中心模块演进计划第四波：工作台驾驶舱增强

日期：2026-04-05

状态：已完成

## 1. 第四波目标

第四波聚焦“工作台驾驶舱增强”，目标不是把 `/workbench` 做成另一套任务页或消息页，而是让它成为四模块收敛之后的高价值个人入口。

本波目标为：

1. 让工作台优先表达“现在先处理什么”，而不是堆放弱业务指标。
2. 去掉与个人驾驶舱弱相关的统计卡片和低行动性的趋势图。
3. 把任务、日程、消息、预警四类信息收敛为可执行的优先队列与风险入口。
4. 让工作台聚合读模型与前三波已完成的 canonical owner 保持一致，尤其是共享日程可读边界。

## 2. 变更分类

本波分类如下：

1. `business model or domain redesign`
2. `requirement-gap`
3. `rule-conflict`

根因不是工作台没有数据，而是它此前同时存在以下问题：

1. 顶部卡片混入“预算总额”“中标金额”这类弱驾驶舱指标。
2. “我的任务”统计混算预警通知，违背第二波已锁定的任务中心口径。
3. 工作台日程聚合仍只看 owner，未对齐第三波共享可读日程边界。
4. 项目趋势图和重复快捷入口占据空间，但不能直接回答“今天先做什么”。

## 3. 第四波定版结论

### 3.1 Canonical 页面与 API

1. `/workbench` 仍是个人驾驶舱 canonical 页面。
2. `/api/workbench/summary` 仍是工作台唯一聚合读模型 API，不承载业务写规则。
3. 任务、日程、消息、预警的写入与详情处理仍分别归属于 `/tasks`、`/calendar`、`/messages`、`/alerts/histories`。

### 3.2 第四波范围边界

本波只做以下五件事：

1. 顶部统计卡片收敛为“待处理事项、我的任务、待处理预警、未读消息、我的项目”。
2. 新增“今日优先队列”，把待办、任务、近期日程收敛为单一行动列表。
3. 新增“风险雷达”，把待处理预警显式提升为驾驶舱主信息。
4. 新增“消息待办”，让工作台直接暴露消息中心中未读的动作性消息。
5. 工作台日程聚合与第三波保持一致，支持 owner 或 participant 可见的共享日程。

本波明确不做：

1. 不把工作台变成新的任务 CRUD 页面。
2. 不把预警处理动作直接内嵌到工作台。
3. 不在本波重构最近动态源或活动流模型。
4. 不在本波解决与工作台无关的其他性能告警或 panel 数据问题。

## 4. 实施要求

1. 工作台只做聚合和分流，不新增平行业务写入口。
2. 任务、预警、消息统计口径必须拆分，不能再通过单一卡片混算。
3. 共享日程在工作台中的读取边界必须与 `/calendar` 一致。
4. 所有工作台入口必须导向已锁定的 canonical owner 页面，而不是重新引入分叉入口。

## 5. 验证面

本波至少验证：

1. `/api/workbench/summary` 正确拆分任务、预警、消息三类统计口径。
2. 工作台优先队列能够同时展示待办、任务、近期日程。
3. 工作台能展示共享给当前用户的日程，而不是只展示 owner 日程。
4. 风险雷达和消息待办能在正式运行态被看到并进入对应 canonical 页面。

## 6. 当前结论

1. `/api/workbench/summary` 已重构为驾驶舱型聚合读模型，新增 `focusQueue`、`riskAlerts`、`unreadMessages`，并拆分 `pendingAlerts`、`unreadMessages` 统计口径。
2. `/workbench` 已移除“我的预算”“我的中标”和弱趋势图/重复快捷入口，改为优先队列、风险雷达、消息待办、重点项目和最近动态的组合。
3. 工作台本周日程已对齐第三波共享读取边界，可展示共享给当前用户的日程并标记“共享”。
4. focused API 回归 `tests/api/workbench/summary-route.test.ts` 已通过 `2/2`，覆盖驾驶舱统计口径拆分、共享日程可见性和安全兜底。
5. focused formal 回归 `tests/e2e/playwright/wave4-formal.spec.ts` 已在 `5004` 端口通过 `1/1`，验证了优先队列、风险雷达、消息待办和共享日程在正式运行态可见。

下一波可以继续评估：

1. 最近动态源按当前用户权限和关注对象进一步收敛。
2. 预警处理动作与任务升级动作之间的更强联动。
3. 工作台与工作日志/报告生成之间的角色边界进一步精简。
