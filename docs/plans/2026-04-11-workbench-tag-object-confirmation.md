# 2026-04-11 工作台标签对象确认单

日期：2026-04-11

文档目标：把“工作台标签收拢到工作台下方”中的“标签对象”先唯一化，避免在目标对象未明确前直接进入界面开发。

## 1. 当前锚点

当前工作台页面已确认存在以下区域：

1. 页面头部快捷按钮：风险雷达、消息中心、工作日志。
2. 工作台模块导航：统一汇总日程管理、任务中心、消息中心、预警管理入口。
3. 驾驶舱指标卡：待处理事项、我的任务、待处理预警、未读消息、我的项目。
4. 下方信息区：今日优先队列、今日待办、本周日程、重点项目、个人事件收件箱。

## 2. 当前问题

经过代码检索，当前没有明确命名为“标签区”的独立组件或独立数据结构。

这意味着当前需求存在一个直接风险：

1. “标签”可能指模块导航入口。
2. “标签”也可能指驾驶舱指标卡。
3. 甚至可能指另一个尚未在现页面中落地的业务入口集合。

若在这个阶段直接开发，极容易把错误对象做迁移。

## 3. 候选对象判断

### 3.1 候选对象 A：工作台模块导航

支持理由：

1. 这是当前工作台最接近“入口集合”的区域。
2. 其内容以模块入口卡片为主，最容易被业务口头描述成“标签”。
3. 若需求表达来自视觉认知，业务方把这组入口称为“标签”是有可能的。

风险：

1. 该区域当前是卡片式模块导航，不是轻量标签式组件。
2. 若按“标签收拢”理解，很可能要先做组件形态重定义，而不只是位置移动。

### 3.2 候选对象 B：驾驶舱指标卡

支持理由：

1. 这些指标卡也是一组首屏快捷入口。
2. 业务方若强调“放到工作台下方”，可能在说首屏上方入口不要占太多空间。

风险：

1. 指标卡承担的是摘要指标语义，不是标签语义。
2. 若把它们当作标签迁移，页面信息层级会被误改。

### 3.3 候选对象 C：尚未落地的业务标签集合

支持理由：

1. 需求单中的“标签”可能来自旧原型、旧版本页面或业务口头认知，而不是当前代码中的现成区域。

风险：

1. 若是这种情况，当前仓内根本没有直接迁移对象。
2. 那么后续应走“新建区域设计”，而不是“现有区域迁移”。

## 4. 当前建议判断

从当前代码看，最可能被业务方称为“标签”的对象，是“工作台模块导航”这一组入口卡片。

但这仍然只是当前最强候选，不应直接视为最终结论。

## 5. 建议确认问题

在进入 ST-04 之前，至少要让业务或产品明确以下 5 个问题：

1. 这里的“标签”是不是当前工作台模块导航这组入口。
2. 若是，它是保持卡片样式，只做位置下移，还是改成轻量标签样式。
3. “工作台下方”具体是页面头部下方、指标卡下方，还是整屏主体区域下方。
4. 标签数量较多时，是折行、横向滑动，还是折叠收起。
5. 标签点击后仍跳原页面，还是改成在工作台内展开对应内容。

## 6. 建议通过标准

若以下 4 项全部确认，则 ST-04 才能视为规则验收通过：

1. 已唯一确认“标签对象”是哪一块区域。
2. 已确认迁移后位置。
3. 已确认迁移后展示形态。
4. 已确认多标签场景和点击行为规则。

## 7. 对开发的直接约束

本确认单未通过前，不建议直接进入前端改造。

原因：

1. 当前问题不是 CSS 或布局微调，而是目标对象未唯一化。
2. 一旦对象判断错误，后续所有前端工作都属于错误实现。

## 8. 统一结论

工作台标签收拢这一项当前仍属于“先定义对象，再做设计”的状态。现阶段最合理的做法不是继续推代码，而是把业务方口中的“标签”唯一指认出来。只有这样，ST-04、ST-05 才有实际落点。
