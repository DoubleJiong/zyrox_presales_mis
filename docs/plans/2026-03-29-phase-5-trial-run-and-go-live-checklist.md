## 2. 阶段边界

### 本阶段要做

1. 选择试点业务线和试运行样本。
2. 组织真实业务流转演练。
3. 对账核心指标、审批链路和权限边界。
4. 收口试运行暴露问题并重新验证。
5. 完成正式投产评审和投产发布。

### 本阶段不做

1. 不重新开启大规模架构重写。
2. 不新增与试运行无关的功能需求。
3. 不在试运行阶段再修改核心统计口径定义。
4. 不在没有对账完成前直接宣布正式投产。

## 3. 进入条件

开始第五阶段前，至少满足：

1. 第四阶段已完成退出标准。
2. 有试运行候选版本。
3. 有试点业务线负责人和业务陪跑责任人。
4. 有明确的试运行问题登记和升级机制。

## 4. 退出标准

阶段结束时，至少满足以下结果：

1. 试点流程可闭环。
2. 核心指标口径与人工样本对账通过。
3. 核心权限边界无阻断级偏差。
4. 阻断问题已收口并复验通过。
5. 正式投产评审通过并完成上线。

补充说明：

1. “试点流程可闭环”在本轮必须先落到“客户/项目/方案”三条可视主链路可进入、可取数、可核对。
2. 若业务账号无法进入客户、项目或方案任一页面链路，视为第五阶段阻断，不得跳过直接做人工对账。

## 6. 任务拆分

### Task 1：选择试点业务线和样本范围

目标：避免一开始就让全组织承担系统成熟度风险。

执行要求：

1. 选择一条代表性强、链路完整但范围可控的业务线。
2. 明确样本项目、样本方案、样本审批单、样本人员。
3. 明确参与角色：业务负责人、产品/流程负责人、技术负责人、陪跑人员。

验收：形成试点说明，不允许“全量直接试”。

### Task 2：编排试运行场景与操作脚本

目标：让试运行不是自由探索，而是按可复验场景执行。

执行要求：

1. 设计标准主流程：登录、客户查看、项目查看、方案查看、审批发起、审批通过/驳回、方案评审、仪表盘核对。
2. 设计异常流程：驳回重提、权限拒绝、数据缺失、回滚观察。
3. 明确每个场景的预期结果和记录方式。

验收：有试运行场景清单和结果记录模板。

### Task 3：执行试运行并做核心指标对账

目标：证明系统输出与业务事实一致。

执行要求：

1. 对账 `totalProjects`、`projectsByStage`、`totalSolutions`、`pendingTasks` 等核心指标。
2. 对账试点样本在系统中的客户归属、项目阶段、审批状态、方案状态与人工记录是否一致。
3. 对不一致项记录根因：口径错误、数据迁移错误、权限过滤错误或展示错误。

验收：形成指标对账记录，不允许只凭页面观感判断正确。

### Task 4：验证权限边界和审批链路

目标：确认系统不会在真实角色环境下越权或漏权。

执行要求：

1. 按角色矩阵验证页面可见范围、数据可见范围和操作权限。
2. 验证审批发起人、审批人、抄送人、管理员的行为边界。
3. 验证密码重置、账号创建、关键导出等敏感操作的审计记录。

验收：形成权限验证记录和审批链路验证记录。

### Task 5：收口试运行问题并复验

目标：让试运行成为投产前最后一轮高价值问题清除，而不是问题堆积池。

执行要求：

1. 对试运行问题按阻断、严重、一般分级。
2. 阻断问题必须修复并复验通过后才能进入投产评审。
3. 严重问题需给出是否接受风险的明确决策。

验收：问题台账闭环，阻断问题为零。

### Task 6：正式投产评审与发布

目标：在明确风险、门禁和责任的条件下完成正式上线。

执行要求：

1. 按正式投产门禁文档逐项检查。
2. 明确发布窗口、责任人、观察时段和回滚阈值。
3. 执行正式投产发布并完成首轮观察。

验收：形成正式投产评审记录、发布记录和稳定观察期关闭记录。

## 7. 风险提醒

1. 如果试点样本不具代表性，试运行结论会失真。
2. 如果只做功能验证、不做指标对账，投产后仍会爆发口径争议。
3. 如果没有问题分级和冻结规则，试运行会退化成持续开发期。
