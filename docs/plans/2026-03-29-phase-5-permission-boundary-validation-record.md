## 6. 2026-03-31 项目闭环前最小依赖清单

### 执行顺序

1. 先闭环项目主链路。
2. 只补齐直接支撑项目闭环的用户、权限、字典依赖，不提前展开整个平台治理。
3. 项目闭环通过后，再单独做“用户管理 / 用户权限 / 字典”正式闭环。

### 用户管理侧最小依赖

1. 项目成员添加、移除、角色调整所依赖的选人链路必须稳定可用。
2. 方案评审人、项目团队成员、项目导入负责人映射都依赖有效用户主数据，至少要保证：
	- 用户搜索接口可返回 `id / realName / email / department`
	- 已删除用户不会继续出现在可选列表中
	- Excel/批量导入时，按姓名映射用户不会大面积失配
3. 项目闭环阶段不需要先完成完整用户后台治理，但必须保证“能被项目/方案选到的人”是稳定的。

### 权限侧最小依赖

1. 项目页列表按钮权限与项目对象级权限要一致，不能出现“前端可点、后端拒绝”或“前端隐藏、后端实际可写”的双轨行为。
2. 项目相关高频写路径至少要复核以下权限边界：
	- 新建项目
	- 编辑项目
	- 删除项目
	- 管理项目成员
	- 编辑商机/招投标/归档信息
	- 新增/删除项目跟进
3. 当前项目接口同时依赖通用权限映射和 `lib/permissions/project.ts` 的对象级判断；项目闭环阶段优先验证真实业务角色在这两层上的结果一致，而不是先重做整套 RBAC 架构。

### 字典侧最小依赖

1. 项目新建/编辑至少依赖以下字典或选项源稳定：
	- `project_status`
	- `priority`
	- `project_type` 或项目类型表
2. 项目导入链路还依赖 `project_status` 字典项名称到编码的映射，必须确认名称与导入模板口径一致。
3. 客户带入项目时，客户区域、客户类型、默认项目类型要能正确回填项目表单；否则项目主链虽然能保存，但业务录入效率和一致性会被反复打断。

### 作为项目闭环前置阻断项的内容

1. 选人失败或成员管理失败。
2. 真实角色在项目页面/接口上出现权限不一致。
3. 项目状态、优先级、项目类型等关键下拉为空、错配或导入映射错误。

### 暂不提前展开的内容

1. 完整用户后台 CRUD 体验治理。
2. 全系统角色矩阵重构。
3. 所有字典类别的全面清洗。

### 当前建议

1. 下一步直接按“项目主链 + 三类最小依赖”推进回归与扫雷。
2. 一旦项目闭环完成，再把用户管理、权限、字典分别作为独立闭环主题收口。

### 当前阻断项

1. 系统预置字典项目前不可在 GUI 中修改，只能新增非系统项；而 `/api/dictionary/sync` 又会把配置文件中的系统字典重新同步回数据库。
2. 项目域仍有多处硬编码映射直接绕过字典：
	- 项目优先级显示与顶部 `PrioritySelector`
	- 项目状态/阶段 label 与颜色映射
	- 项目导入的状态、优先级、行业、区域映射表
3. 项目阶段不是普通字典问题，而是业务状态机问题；即使提供 GUI，也不能让字典直接替代阶段迁移规则和权限约束。
4. 项目字典历史包袱仍然存在：数据库初始化里还有旧 `project_status` 口径（如 `draft / proposal / implementation / completed`），与当前三阶段治理后的真实业务口径并不一致。

### 对后续治理的含义

1. 可以做成“GUI 维护业务字典，代码只负责状态机和样式扩展”的模式。
2. 但项目域至少要先完成三件事：
	- 明确哪些属于“可配置业务字典”，哪些属于“不可随意改的流程状态机”
	- 把项目页面和导入链路中的硬编码 label / option 映射收敛到统一字典读取层
	- 调整系统字典策略，避免 GUI 改完后又被配置同步覆盖
3. 完成这三步后，业务和技术实现才算真正整合，而不是“有 GUI，但核心口径仍埋在代码里”。

## 11. 2026-03-31 字典平台收口边界

### 当前代码事实

1. 运行时下拉组件 `DictSelect` 已统一通过 `/api/dictionary/options` 从数据库取值，这说明“GUI 改值 -> 页面下拉生效”在技术上已经成立。
2. 但当前 GUI 只能新增、修改、删除 `isSystem=false` 的分类和字典项；系统预置项在 `/api/dictionary/items` 和 `/api/dictionary/categories` 中仍被明确禁止修改和删除。
3. `/api/dictionary/sync` 仍会把 `dictionary-config.ts` 里的系统分类和系统项重新同步回数据库，因此即使绕过前端限制直接改系统字典，也存在被配置同步覆盖的风险。
4. 项目三阶段模型和项目状态模型并不属于普通下拉：`projectStage`、`projectStatus` 的转换矩阵、可达关系、确认逻辑和附带字段都在 `status-transitions.ts` 与项目接口中以状态机方式实现。

### 一类：可交给 GUI 的业务字典

1. 这类值只负责“选项集合”和“显示文案”，不承载阶段迁移、审批副作用或权限判断，原则上应交给业务侧通过 GUI 维护。
2. 当前项目域中最适合纳入这一类的包括：
	- `industry`（客户类型口径）
	- `region`
	- `project_priority`
	- `bidding_method`
	- `scoring_method`
	- `fund_source`
	- `bid_type`
	- `bond_status`
	- `archive_status`
3. 这些类别的治理目标应是：页面录入、筛选、导入导出、报表展示全部只消费数据库中的当前字典项，不再保留并行硬编码映射。

### 二类：应独立成业务主数据，不建议直接塞进通用字典

1. 这类值虽然看起来像枚举，但实际已经被业务表或外键引用，后续更适合做成“独立主数据表 + 专用 GUI”而不是继续混在通用字典项里。
2. 当前已被代码验证的重点对象：
	- `project_type`，真实权威源应为 `sys_project_type`
	- 客户类型默认项目类型的关联能力，当前已落在 `sys_customer_type.default_project_type_code`
	- 售前服务类型，当前已有独立表 `sys_presales_service_type`
3. 这类对象后续可以提供 GUI，但 GUI 应直接维护对应主数据表，而不是再通过 `sys_attribute` 间接映射一次。

### 三类：必须保留为状态机或策略枚举

1. 这类值一旦允许业务直接改字典，就会破坏流程可达性、审批副作用或权限边界，不应视为普通字典维护。
2. 当前明确属于这一类的包括：
	- `project_stage`，受三阶段模型和 tab 访问约束控制
	- `project_status`，受状态转换矩阵、确认弹窗和附加字段约束控制
	- `solution_status`，受评审/审批流驱动，不只是展示标签
	- 项目成员角色等权限敏感枚举，不能允许业务自由增删改后直接进入鉴权判断
3. 这类字段可以在 GUI 中展示说明和映射文案，但其“可取值集合、迁移规则、约束条件”必须继续由代码中的状态机和策略层控制。

### 收口建议

1. 先把通用字典平台拆成两种能力：
	- 业务字典维护
	- 状态机/系统枚举展示
2. 对业务字典，应取消“系统项永不可改但又要业务自己维护”的矛盾设定，至少要解决以下二选一：
	- 允许系统字典在受控权限下直接编辑，并停止 `/api/dictionary/sync` 覆写运行中数据
	- 保留配置同步，但把真正需要业务维护的类别迁出 `isSystem=true` 体系
3. 对主数据表型枚举，应逐步从通用 `sys_attribute` 撤出，改成专用管理页，避免继续出现“双权威源”问题。
4. 对状态机型枚举，应明确标注“只读解释，不可配置”，避免业务误以为改字典就能改流程。
