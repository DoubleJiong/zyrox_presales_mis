# 工作台、日程、任务、消息中心模块演进计划第一波：定位与目标设计

日期：2026-04-04

状态：已完成

## 1. 背景与目标

本轮工作不再把“工作台、日程管理、项目任务、消息中心”视为四个孤立页面，而视为一个协同执行面。

当前系统这四块已经各自具备一定能力，但整体仍存在明显的角色重叠、入口分裂、权限边界不稳和事件来源不统一问题。

本计划第一波的目标不是直接堆功能，而是先把以下内容定版：

1. 四个模块各自应承担什么角色。
2. 哪些入口、路由、统计和事件是 canonical owner。
3. 哪些现有实现属于 split-brain 或弱治理实现，必须在后续波次中收敛。
4. 第一批开发波次应该先收哪条线，才能既提升业务价值，又不把系统带偏。

## 2. 任务分类与治理依据

### 2.1 任务分类

本轮任务归类为：

1. `business model or domain redesign`
2. `requirement-gap`
3. `rule-conflict`

它不是单点 bug 修复，也不是局部 UI 调整。

### 2.2 治理依据

本轮设计以以下文档为上位约束：

1. `docs/plans/2026-03-29-production-hardening-and-business-refactor-plan.md`
2. `docs/plans/2026-03-29-rebuild-vs-upgrade-assessment.md`
3. `docs/plans/2026-03-29-project-state-machine-and-approval-system-design.md`
4. `ISSUE_SUMMARY_v1.md`

与这些治理文档保持一致的原则如下：

1. 继续采用受控升级，不新开平行系统。
2. 优先收敛业务边界，而不是继续叠加零散功能。
3. 避免新增长期并行入口、并行 API、并行消息来源、并行统计口径。
4. 所有后续实现都必须走正式构建、正式验证、正式上线口径，而不是以 dev 模式代替发布验证。

## 3. 当前现状判断

### 3.1 工作台

当前 `workbench` 已具备聚合价值，能汇总待办、日程、重点项目、动态和个人统计，但它本质上仍是一个聚合读模型，而不是独立业务中枢。

当前问题：

1. 聚合信息较多，但动作闭环主要仍跳转到其他模块完成。
2. “我的任务”统计混合了预警通知和项目任务，业务语义已经超出页面文案表达。
3. 工作台承载了个人协作入口，但其下游任务、日程、消息三条线还没有统一事件定义。

### 3.2 日程管理

当前系统同时存在 `/calendar` 与 `/schedules` 两条体验线；导航把“日程管理”指向 `/calendar`，而 `/schedules` 仍作为独立 CRUD 页面存在。

当前问题：

1. 信息架构分裂，用户难以理解哪个才是 canonical 日程入口。
2. 日程目前主要是用户自有记录，协作参与人还是自由文本，尚未形成可追踪的团队协同机制。
3. 待办和日程在日历页已做展示聚合，但 domain owner 仍分散。

### 3.3 任务管理

当前 `/tasks` 已经是四个模块中业务性最强的一块，具备项目权限过滤、任务 CRUD、负责人、优先级、状态和项目关联。

当前问题：

1. 导航名称仍是“项目任务”，但系统同时还有 `/my-tasks`、`todos`、预警任务，用户对“任务”的心智被拆散。
2. `/tasks` 与 `/my-tasks` 在业务上并非两套 domain，只是两种视角，却被拆成两套页面与 API 聚合。
3. 任务与日程已有联动，但规则仍偏附带逻辑，没有形成统一的执行对象模型。

### 3.4 消息中心

当前消息中心 UI 已经具备 inbox 形态，但业务闭环和治理成熟度最弱。

当前问题：

1. `/api/messages` 及其配套路由仍依赖显式 `userId` 参数，而不是从认证态绑定当前接收人。
2. 消息类型、提醒、通知、动作链接虽然存在，但来源治理不清，仍更像收件箱，不像业务事件中枢。
3. 已读、全部已读、未读数等动作缺少足够强的 receiver ownership 约束。

## 4. 第一波定版后的模块角色

### 4.1 工作台：个人执行驾驶舱

工作台的角色应明确为：

1. 个人视角的执行驾驶舱。
2. 负责展示“今天要处理什么、哪些风险要优先看、哪些项目值得立即进入”。
3. 只做高价值聚合与快速分流，不承载复杂维护流程本体。

工作台不应该成为：

1. 另一套任务管理页。
2. 另一套消息中心。
3. 另一套完整日程管理页。

### 4.2 日程管理：时间与承诺管理层

日程模块的角色应明确为：

1. 个人与协作型时间安排的管理层。
2. 以 `/calendar` 作为 canonical 入口。
3. 负责承接待办时间化、会议安排、拜访安排、提醒计划等时间维度动作。

日程模块不应该继续维持：

1. `/calendar` 与 `/schedules` 长期并行的双入口结构。
2. 仅靠自由文本参与人维持“协作日程”假象。

### 4.3 任务管理：执行对象主中心

任务模块的角色应明确为：

1. 负责承接可指派、可跟踪、可推进的执行事项。
2. 以 `/tasks` 为 canonical 任务中心。
3. `/my-tasks` 应逐步收敛为 `/tasks` 内的“我的任务”视角，而不是长期独立域。

任务模块暂不直接吞并：

1. 轻量个人待办 `todo`
2. 纯消息提醒

但它必须成为这些对象的上游或下游联动中心，而不是并列碎片之一。

### 4.4 消息中心：认证绑定的事件收件箱

消息中心的角色应明确为：

1. 面向当前登录用户的事件收件箱。
2. 负责接收系统通知、业务提醒、动作触发型消息。
3. 所有读取、已读、删除、未读计数都必须以当前认证用户为边界。

消息中心不应被定义为：

1. 即时通讯工具。
2. 任意用户可伪造 `userId` 拉取的通知仓库。
3. 替代任务或审批流的平行工作流。

## 5. Canonical Owner 定版

### 5.1 页面与入口

1. `workbench` 是个人驾驶舱入口。
2. `calendar` 是日程管理 canonical 页面。
3. `tasks` 是执行任务 canonical 页面。
4. `messages` 是个人事件收件箱 canonical 页面。

### 5.2 API 与读模型

1. `/api/workbench/summary` 仅作为工作台聚合读模型，不承载新业务写规则。
2. `/api/tasks` 作为项目任务主 API；后续“我的任务”应尽量通过同一 domain 能力收敛视图，而不是长期维持双主入口。
3. `/api/schedules` 作为日程 domain API，但其体验入口向 `/calendar` 收敛。
4. `/api/messages` 及其配套路由必须收敛为 auth-bound receiver model。

### 5.3 事件与联动

后续演进中应形成以下上游关系：

1. 任务负责执行推进。
2. 日程负责时间承诺。
3. 消息负责事件送达。
4. 工作台负责个人优先级排序与分流。

不能再继续出现以下平行语义：

1. 同一个提醒既像消息又像任务又像工作台统计但没有主归属。
2. “我的任务”和“项目任务”由不同页面长期分别定义。
3. 日历页已经是主入口，但单独日程页仍继续演化为另一套主入口。

## 6. 当前结构冲突清单

本轮确认的结构冲突如下：

1. `calendar` 与 `schedules` 双入口冲突。
2. `tasks`、`my-tasks`、`todos`、预警任务之间存在任务心智分裂。
3. `messages` 相关 API 仍未完全走 auth-bound 模式。
4. 工作台统计混合多种执行对象，但其下游 owner 尚未完全统一。
5. 导航命名仍偏“项目任务”，但实际业务已接近更广义的执行中心。

## 7. 交付波次拆解

### 7.1 第一波

第一波范围定义为：入口收敛与信任边界收敛。

目标：

1. 锁定四个模块角色。
2. 锁定 canonical 页面与 API owner。
3. 消除最危险的信任边界问题。
4. 为第二波任务中心收敛和第三波日程协作升级打基础。

本波推荐实现项：

1. 收敛消息中心 API 到 `withAuth` 当前用户边界。
2. 修正消息未读数、全部已读、单条已读、删除等动作的 receiver ownership。
3. 明确导航与文案，把 `/tasks` 朝“任务中心”语义推进，但不在本波强行吞并所有待办对象。
4. 明确 `/calendar` 为日程主入口，并把 `/schedules` 逐步降级为日程维护子路径或后续待移除入口。
5. 保持工作台只做聚合读模型，不在本波把复杂维护动作继续堆回工作台。

### 7.2 第二波

第二波范围定义为：任务中心视角收敛。

目标：

1. 收敛 `/tasks` 与 `/my-tasks`。
2. 清理任务统计和任务分类口径。
3. 明确项目任务、预警任务、个人待办的边界和联动。

### 7.3 第三波

第三波范围定义为：日程协作能力升级。

目标：

1. 收敛 `calendar` 与 `schedules`。
2. 把参与人、提醒、重复规则从弱文本配置升级到受控结构。
3. 建立日程与任务、消息的联动闭环。

### 7.4 第四波

第四波范围定义为：工作台驾驶舱增强。

目标：

1. 让工作台真正体现个人优先级和风险排序。
2. 去掉重复卡片和弱业务指标。
3. 让工作台成为四模块收敛后的高价值入口，而不是杂糅看板。

## 8. 第一波验证面

第一波至少需要验证以下内容：

1. 消息中心所有读写动作只能作用于当前认证用户。
2. `/messages` 页面在正式构建产物上可正常拉取、已读、全部已读、删除。
3. 导航与页面文案调整后，不会引入死入口。
4. `/calendar` 和现有日程维护路径之间的跳转仍可用。
5. 工作台相关聚合 API 未因入口收敛而失真。

## 9. 非目标

第一波明确不做以下事项：

1. 不在本波把 `todo`、项目任务、预警任务彻底并表。
2. 不在本波把消息中心扩展为 IM 或评论系统。
3. 不在本波引入新的复杂协作日程审批流。
4. 不在本波把工作台重做成重交互大型 cockpit。

## 10. 第一波完成标准

当以下条件同时满足时，可视为第一波闭环：

1. 四个模块的角色、入口和 owner 已通过本文档定版。
2. 消息中心信任边界已收口到 auth-bound 模式。
3. 导航与入口收敛方向已通过代码体现，且无死入口。
4. 正式构建、相关测试与人工验证完成。
5. 可无歧义进入第二波任务中心收敛。

## 11. 当前结论

当前最适合作为第一波落地点的，不是直接重做工作台，而是先处理：

1. 消息中心信任边界
2. 任务与日程入口语义收敛
3. 工作台读模型边界固化

这是风险最低、收益最高、且最有利于后续持续自动推进的切入顺序。
