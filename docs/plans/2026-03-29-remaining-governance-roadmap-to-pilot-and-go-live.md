# 从当前状态到试运行与正式投产的剩余治理路线图

日期：2026-03-29

目标边界：以“已完成试运行并正式投产”为治理完成标准。

适用前提：

1. 第一阶段已完成并通过基线验证。
2. 第二阶段核心定版文档已建立。
3. 第三阶段尚未进入代码改造批次。

## 1. 结论

距离“整个系统治理完成”，除第二阶段和第三阶段的剩余工作外，仍有明确且不可跳过的后续工作。至少包括：

1. 第四阶段：测试体系、发布体系、监控与运维硬化。
2. 第五阶段：试点业务线试运行、真实样本对账、正式投产评审。
3. 跨阶段生产验收：业务、工程、安全、运维四类门禁同时满足。

因此，当前完整收尾路径应为：

1. 第二阶段收口
2. 第三阶段落地
3. 第四阶段工程硬化
4. 第五阶段试运行与正式投产
5. 正式投产后稳定观察期关闭

## 2. 当前状态判断

当前系统已具备进入下一轮治理的基础，但还不具备直接正式投产条件。主要原因：

1. 业务真相已完成代码与数据库层收敛，但仍需人工审核确认“实现正确”与“业务接受”。
2. 审批、状态机、方案主模型、仪表盘口径与身份生命周期统一实现已完成，当前剩余的是真实角色人工复核与发布后稳定观察关闭。
3. 已有最小 CI、正式 SQL 迁移、回滚清单、定向 preflight 与备份路径，投产级交付链路已基本闭环。
4. 当前不直接宣布正式投产的原因，已经从“工程能力缺失”转为“需要完成人工审核、发布窗口确认和观察期关闭”。

## 3. 剩余阶段全景

### 3.1 第二阶段剩余工作

目标：把现有定版文档收口为唯一治理输入，禁止后续实现重新发散。

剩余动作：

1. 确认术语表、状态机、审批模型、方案主模型、统计口径之间不存在相互冲突。
2. 为第三阶段每个 slice 补足“输入文档 -> 改造边界 -> 验收口径”的映射。
3. 形成第二阶段退出评审记录，明确允许进入第三阶段的前提。

退出物：

1. 第二阶段退出评审记录
2. 第三阶段批次计划定版
3. 统一术语与统计口径引用基线

### 3.2 第三阶段剩余工作

目标：把第二阶段的统一设计真正落入代码、数据结构、接口和测试。

执行顺序：

1. Slice 1：项目状态机服务
2. Slice 2：统一审批实体
3. Slice 3：方案评审与方案主模型收敛
4. Slice 4：仪表盘与统计口径切换
5. Slice 5：身份生命周期治理

第三阶段完成后应达到：

1. 核心业务主链路不再靠 route 即时拼装。
2. 核心业务对象有统一真相源。
3. 关键流程有服务层、策略层、数据访问层边界。
4. 关键迁移有兼容策略和回归验证。

### 3.3 第四阶段新增工作

目标：让系统从“能运行的重构后版本”进入“可重复交付、可监控、可回滚、可演练”的工程状态。

核心主题：

1. 测试金字塔补齐
2. CI/CD 与发布门禁升级
3. 日志、监控、告警建立
4. 容器化与环境标准化
5. 备份、恢复、回滚、运行手册成型

这是第二、三阶段之外新增且必须完成的工作，不能用“代码已经重构好”替代。

### 3.4 第五阶段新增工作

目标：在真实业务样本和真实操作人员环境下确认系统可信，再决定正式投产。

核心主题：

1. 选择试点业务线
2. 试运行演练与陪跑
3. 数据与统计口径对账
4. 权限边界与审批流验证
5. 缺陷收口与正式投产评审

这同样是第二、三阶段之外新增且必须完成的工作，不能用“测试通过”替代。

## 4. 建议执行节奏

### 批次 A：第二阶段收口

目标：锁定第三阶段输入。

完成标准：

1. 第二阶段退出评审通过。
2. 第三阶段每个 slice 的输入和验收标准明确。

### 批次 B：第三阶段 Batch A

范围：

1. Slice 1：项目状态机服务骨架
2. Slice 2：统一审批实体骨架

完成标准：

1. 项目主阶段迁移走统一服务。
2. 审批主表与事件骨架可用。

### 批次 C：第三阶段 Batch B

范围：

1. Slice 2：投标立项审批接入
2. Slice 3：方案评审接入统一审批

完成标准：

1. 审批不再等于直接改项目状态。
2. 方案评审不再绕开统一审批事件。

### 批次 D：第三阶段 Batch C

范围：

1. Slice 4：仪表盘口径切换
2. Slice 5：身份生命周期治理

完成标准：

1. 核心指标来源与业务真相一致。
2. 人员与账号生命周期形成审计闭环。

### 批次 E：第四阶段工程硬化

完成标准：

1. 自动化测试达到发布门槛。
2. 发布、回滚、备份、恢复、监控、告警可演练。
3. 有稳定部署与观察机制。

### 批次 F：第五阶段试运行与正式投产

完成标准：

1. 试点流程闭环。
2. 关键指标对账通过。
3. 权限边界与审批链路通过真实验证。
4. 正式投产评审通过并完成首轮稳定观察。

## 5. 各阶段关键依赖

1. 未完成第二阶段收口，不进入第三阶段代码实施。
2. 未完成第三阶段核心收敛，不进入第四阶段发布硬化验收。
3. 未完成第四阶段运维与监控准备，不进入第五阶段试运行。
4. 未完成第五阶段试运行对账，不进入正式投产。

补充更新（2026-04-04）：上述依赖链当前已全部满足；剩余 gate 不再是阶段性工程前置，而是人工审核与稳定观察项关闭。

## 7. 投产前必须新增的治理能力

结合当前仓库现状，以下能力不能假设已经具备，必须显式建设并验证：

1. 标准化部署产物，而不是依赖本地开发机运行方式。
2. 容器化或等效环境标准化方案，而不是停留在文档示例。
3. 错误监控、接口日志、业务审计和告警规则。
4. 备份恢复实操演练，而不是只有环境变量占位和页面入口。
5. 运行手册、值守责任、故障升级路径和回滚纪律。

## 8. 正式投产判定

只有在以下条件全部满足时，才建议视为“治理完成并可正式投产”：

1. 第二阶段和第三阶段的业务与模型收敛全部落入代码与数据结构。
2. 第四阶段发布与运维硬化全部完成并经演练验证。
3. 第五阶段试运行完成，业务流程、统计口径、权限边界均通过对账。
4. 正式投产门禁文档中的业务、工程、安全、运维四类条件同时满足。
