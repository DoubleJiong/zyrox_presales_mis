# 售前管理系统方案模型收敛策略

**目标：** 为 `scheme` 与 `solution` 双模型并存问题给出第二阶段正式结论，防止第三阶段继续在两套模型上同时扩展。

---

## 1. 结论先行

第二阶段结论：

1. `solution` 作为未来唯一主方案模型。
2. `scheme` 视为历史兼容模型，不再承载新的主能力扩展。
3. 现有仍直接依赖 `scheme` 的页面、统计、权限与仪表盘，在第三阶段按“兼容读取 -> 主实现迁移 -> 旧入口退场”路线处理。

## 2. 当前系统现状

当前仓库至少同时存在两套方案相关结构：

1. `bus_scheme`、`bus_scheme_file`、`bus_scheme_template`
2. `bus_solution`、`bus_solution_version`、`bus_solution_sub`、`bus_solution_review`、`bus_solution_statistics`

同时，当前运行实现仍存在以下问题：

1. `api/dashboard` 仍统计 `schemes`，而不是以 `solution` 为主来源。
2. `solution-review.service.ts` 虽已基于 `solution` 体系工作，但其评审完成后只回写局部状态，没有形成统一审批事件模型。
3. 角色字典与命名规范已明确未来主资源名为 `solution`，但代码层尚未完成同步。

## 3. 模型定位

### 3.1 `solution`

未来主模型，承载以下能力：

1. 方案主记录
2. 版本管理
3. 子方案结构
4. 方案成员与权限
5. 方案评审
6. 方案统计与下载/使用记录

### 3.2 `scheme`

历史兼容模型，第二阶段仅允许承担以下职责：

1. 兼容旧页面读取
2. 兼容旧仪表盘与旧 API 过渡统计
3. 为第三阶段迁移提供历史源数据

第二阶段明确禁止：

1. 再新增基于 `scheme` 的新业务主流程
2. 再新增只写 `scheme` 不写 `solution` 的新能力
3. 再新增以 `scheme` 为唯一统计来源的新仪表盘指标

## 4. 收敛路线

### 阶段 A：口径冻结

1. 所有新设计文档统一使用 `solution` 作为主术语。
2. 所有权限 key、policy 命名、接口语义统一面向 `solution`。

### 阶段 B：读取兼容

1. 允许短期保留旧接口读取 `scheme`。
2. 任何新统计与新评审逻辑必须优先从 `solution` 体系取数。

### 阶段 C：主写入切换

1. 新建、编辑、评审、版本、成员、统计统一写入 `solution` 体系。
2. `scheme` 仅保留迁移前历史读取与回放责任。

### 阶段 D：退场准备

1. 梳理仍依赖 `scheme` 的页面与接口清单。
2. 为历史数据迁移与兼容层下线建立第三阶段切片。

## 5. 第三阶段实施顺序建议

1. 先替换统计和仪表盘来源，停止继续扩大 `scheme` 的口径影响。
2. 再替换方案评审和权限判断，使其只围绕 `solution`。
3. 再梳理模板、文件和关联项目能力。
4. 最后做数据迁移与旧入口退场。

## 6. 验收判断

当以下条件满足时，可认为第三阶段的方案模型收敛实施完成：

1. 新增方案主流程全部基于 `solution`
2. 仪表盘不再把 `scheme` 作为主统计来源
3. 评审、权限、统计和下载审计都以 `solution` 体系为主
4. `scheme` 仅剩兼容读取或已完成退场
