# 工作台、日程、任务、消息中心模块演进计划第五波：最近动态源收敛

日期：2026-04-05

状态：已完成

## 1. 第五波目标

第五波聚焦“最近动态源收敛”，目标是把工作台中的最近动态从匿名、全局、弱过滤的数据流，收敛为认证绑定且与当前用户相关的 activity feed。

本波目标为：

1. 让 `/api/activities` 从匿名接口收口到当前认证用户边界。
2. 让最近动态只展示与当前用户直接相关，或与其可访问项目相关的活动。
3. 让工作台最近动态与既有项目权限模型保持一致，而不是重新定义一套可见性规则。
4. 保持 `/workbench` 作为最近动态唯一消费入口，不新增新的活动页面或并行数据源。

## 2. 变更分类

本波分类如下：

1. `business model or domain redesign`
2. `security or auth hardening`
3. `rule-conflict`

根因不是工作台组件不会渲染，而是活动流此前存在以下结构问题：

1. `/api/activities` 未走 `withAuth`，匿名即可读取最近动态。
2. 商机、项目、跟进、报价都按“全局最近”取数，未对齐 current-user cockpit 语义。
3. 项目与报价动态没有复用现有项目权限口径，存在看板级别数据泄露风险。

## 3. 第五波定版结论

### 3.1 Canonical 页面与 API

1. `/workbench` 仍是最近动态的唯一 canonical 页面入口。
2. `/api/activities` 仍是工作台活动流的唯一读模型 API。
3. 项目权限口径继续以 `getAccessibleProjectIds()` / `isSystemAdmin()` 为准，不新增活动流专属权限模型。

### 3.2 第五波范围边界

本波只做以下四件事：

1. `/api/activities` 接入 `withAuth`。
2. 商机动态对非管理员收敛为本人 owner 的商机活动。
3. 项目与报价动态对非管理员收敛为当前用户可访问项目范围内的活动。
4. 跟进记录对非管理员收敛为本人完成的跟进动态。

本波明确不做：

1. 不扩展新的活动类型。
2. 不重做工作台最近动态 UI。
3. 不引入独立活动中心页面。
4. 不在本波处理 activity feed 之外的 panel 数据或其他工作台残留问题。

## 4. 实施要求

1. 最近动态必须走认证绑定边界，不得再允许匿名读取。
2. 活动流的项目相关可见性必须复用现有项目权限工具，而不是新建一套活动权限判断。
3. 工作台继续只做消费和展示，不在页面侧拼装权限规则。

## 5. 验证面

本波至少验证：

1. `/api/activities` 未登录访问返回 `401`。
2. `/api/activities` 非管理员只看到本人商机、本人跟进和可访问项目/报价活动。
3. `/workbench` 最近动态在正式 `5004` 运行态仍可看到当前用户相关活动。

## 6. 当前结论

1. `/api/activities` 已切换为 `withAuth` 保护，不再允许匿名获取最近动态。
2. 非管理员现在只会看到本人 owner 的商机、本人跟进，以及位于可访问项目集合内的项目/报价活动；管理员仍保留全局视角。
3. 工作台最近动态消费链路无需新增页面改动，即可继承新的 auth-bound 活动源。
4. focused API 回归 `tests/api/activities/route.test.ts` 已通过 `2/2`，覆盖当前用户 / 可访问项目过滤以及空项目集场景。
5. focused formal 回归 `tests/e2e/playwright/wave5-formal.spec.ts` 已在 `5004` 端口通过 `1/1`，验证了匿名访问被拒绝，且登录后工作台可见当前用户相关活动。

下一波可以继续评估：

1. 最近动态是否需要接入任务、消息、预警等更多 canonical 事件源。
2. 最近动态排序是否需要从“最近创建”进一步升级为“最近相关动作”。
3. 工作台与独立事件收件箱、预警中心之间的长期职责边界是否需要继续压缩。
