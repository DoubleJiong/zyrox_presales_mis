# 第四阶段监控与告警基线

日期：2026-03-29

## 1. 目标

建立最小可运行的可观测性基线，使系统具备“存活可探测、数据库可探测、错误可归类、关键业务动作可审计、告警可升级”的基础能力。

## 2. 健康检查入口

1. `GET /api/health`：存活检查，返回应用状态、时间戳、运行时长。
2. `GET /api/health/ready`：就绪检查，返回数据库状态和连接池摘要。

## 3. 核心监控指标

### 技术指标

1. 接口错误率：按 5 分钟窗口统计，超过 5% 触发 P1。
2. 登录失败率：按 15 分钟窗口统计，超过 20% 触发 P2。
3. 接口耗时：关键接口 P95 超过 1500ms 触发 P2。
4. readiness 失败：连续 3 次失败触发 P1。

### 业务指标

1. 审批提交失败数
2. 阶段迁移失败数
3. 密码重置失败数
4. 关键导出失败数

## 4. 日志与审计最小集合

必须保留并可检索的事件：

1. 登录成功 / 登录失败
2. 审批发起 / 审批处理
3. 项目阶段迁移
4. 用户密码修改 / 重置
5. 关键导出与批量操作

现有落点：

1. `loginLogs`
2. `operationLogs`
3. `alertRules`
4. `alertHistories`
5. `alertNotifications`

## 5. 告警升级路径

1. P1：发布负责人 + 技术负责人，15 分钟内响应。
2. P2：模块负责人，30 分钟内响应。
3. P3：纳入次日排障清单。

## 6. 故障定位入口

1. 先访问 `/api/health` 与 `/api/health/ready`。
2. 再检查应用构建版本、数据库连通性、关键日志表。
3. 如为业务链路问题，优先检查审批、项目阶段和身份生命周期相关审计记录。

## 7. 结论

本基线解决的是“没有统一监控入口”的问题。下一阶段试运行仅允许在此基础上追加指标，不再回退到纯人工感知故障。
