# 2026-04-11 主干需求研发任务卡

日期：2026-04-11

文档目标：把下一轮主干需求 `2.1-2.4` 继续转成可直接录入项目管理系统的研发任务卡。

适用方式：

1. 每张任务卡可直接对应 1 个 issue 或 1 个研发任务单。
2. 若团队人数较少，可把同阶段的相邻任务卡合并执行。
3. 默认按“先交互骨架，后数据口径，最后联调回归”的顺序推进。

## 1. 使用原则

### 卡片状态建议

- `todo`：未开始
- `doing`：开发中
- `review`：待评审 / 待联调
- `qa`：待测试
- `done`：已完成

### 优先级建议

- `P0`：本轮主干必做
- `P1`：建议本轮完成
- `P2`：可顺延

### 预估方式建议

- `XS`：0.5 天以内
- `S`：1 天
- `M`：2 至 3 天
- `L`：4 至 5 天

## 2. 建议建卡顺序

建议按以下顺序录入任务系统：

1. TC-01 至 TC-03

## 3. 任务卡明细

## TC-01 下钻交互模型与入口统一

**任务类型：** 前端 / 产品协同

**优先级：** P0

**建议工期：** M

**建议负责人：** 前端

**前置依赖：** 无

**任务目标：** 为 `2.1` 下钻功能建立统一入口、统一交互规则和统一命名口径。

**涉及范围（预计）：**

1. 项目管理列表页。
2. 项目详情联动入口。
3. 相关卡片或快捷查看入口。

**交付物：**

1. 下钻入口清单。
2. 统一的点击触发规则。
3. 下钻入口命名和按钮口径。

**完成定义：**

1. 用户可从主要页面稳定进入下钻视图。
2. 同类入口不再出现“有的能点、有的不能点”的割裂表现。
3. 产品口径和页面交互说明一致。

**验收要点：**

1. 至少覆盖列表、详情、卡片三个入口。
2. 点击后路径稳定，无误跳或无反馈情况。

## TC-02 下钻详情容器与返回路径

**任务类型：** 前端

**优先级：** P0

**建议工期：** M

**建议负责人：** 前端

**前置依赖：** TC-01

**任务目标：** 完成下钻详情容器、关闭机制、返回路径和上下文保持。

**涉及范围（预计）：**

1. 详情抽屉或详情页容器。
2. 返回上一层或关闭下钻动作。
3. 筛选条件与滚动位置保持。

**交付物：**

1. 统一下钻容器。
2. 返回路径规则。
3. 上下文保持逻辑。

**完成定义：**

1. 下钻后能正常返回原页面。
2. 返回后筛选条件和主要浏览位置不丢失。
3. 不同入口进入的详情体验保持一致。

**验收要点：**

1. 覆盖抽屉型和页面跳转型两类场景。
2. 返回后无明显页面闪动或状态重置问题。
