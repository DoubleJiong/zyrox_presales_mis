# 工作台、日程、任务、消息中心模块演进计划第六波：最近动态接入任务、消息、预警

日期：2026-04-05

状态：已完成

## 1. 第六波目标

第六波聚焦“最近动态接入任务、消息、预警”，目标是把第五波已经完成 auth-bound 收口的 `/api/activities`，继续从“项目相关动态流”推进为真正的个人事件入口。

本波目标为：

1. 让最近动态补齐任务、消息、预警三类 canonical 事件源。
2. 让 `/workbench` 最近动态与第四波驾驶舱中的“今日优先队列 / 风险雷达 / 消息待办”形成前后呼应，而不是继续停留在独立卡片聚合。
3. 保持任务、消息、预警各自页面仍是 canonical owner，工作台只做个人事件入口和分流。

## 2. 变更分类

本波分类如下：

1. `business model or domain redesign`
2. `rule-conflict`
3. `requirement-gap`

根因不是工作台没有展示区域，而是前五波之后仍存在以下结构断层：

1. 工作台 summary 已经有任务、预警、消息聚合，但最近动态仍主要停留在商机、项目、跟进、报价四类项目侧活动。
2. 最近动态与“个人驾驶舱”语义不完全一致，缺少用户此刻最相关的收件箱型事件。
3. 任务、消息、预警虽然都已有 canonical owner 页面，但没有进入统一的个人事件入口。

## 3. 第六波定版结论

### 3.1 Canonical 页面与 API

1. `/workbench` 仍是个人驾驶舱 canonical 页面。
2. `/api/activities` 仍是工作台最近动态的唯一读模型 API。
3. `/tasks`、`/messages`、`/alerts/histories` 继续分别承接任务、消息、预警的详情处理与后续动作。

### 3.2 第六波范围边界

本波只做以下四件事：

1. `/api/activities` 接入当前用户任务动态。
2. `/api/activities` 接入当前用户未读消息动态。
3. `/api/activities` 接入当前用户待处理预警动态。
4. `/workbench` 最近动态补齐三类事件的图标与跳转口径。

本波明确不做：

1. 不改任务、消息、预警自己的写规则或处理状态机。
2. 不引入新的事件中心页面。
3. 不在本波升级最近动态排序算法到“最近相关动作”。
4. 不在本波做预警自动升级任务或更强自动闭环。

## 4. 实施要求

1. 新接入的任务、消息、预警事件必须继续遵守当前用户边界，不得回退到全局读取。
2. 工作台最近动态只负责显示和跳转，不新增并行处理入口。
3. 新事件源接入后，返回结构应保持与现有工作台消费兼容。

## 5. 验证面

本波至少验证：

1. `/api/activities` 现在能够返回 `task`、`message`、`alert` 三类动态。
2. 新三类动态仍然遵守当前用户边界。
3. `/workbench` 最近动态在正式 `5004` 运行态能够展示任务、消息、预警三类新增事件。

## 6. 当前结论

1. `/api/activities` 已接入 `task`、`message`、`alert` 三类事件源，并继续保持 auth-bound 最近动态口径。
2. 工作台最近动态现在会把任务中心、消息中心、风险雷达中的个人相关事件统一汇入同一事件列表。
3. `/workbench` 最近动态已补齐三类事件的图标、标题展示与跳转目标，其中任务跳转到 `/tasks?scope=mine`，消息跳转到消息中心或消息自带动作链接，预警跳转到 `/alerts/histories?status=pending`。
4. focused API 回归 `tests/api/activities/route.test.ts` 已通过 `2/2`，覆盖活动源扩展与权限边界。
5. focused formal 回归 `tests/e2e/playwright/wave6-formal.spec.ts` 已在 `5004` 端口通过 `1/1`，验证任务、消息、预警三类动态能在正式工作台最近动态中看到。

下一波可以继续评估：

1. 最近动态排序是否从“最近创建/更新时间”升级为“最近相关动作”。
2. 预警与任务之间是否建立更强的升级/闭环联动。
3. 工作台、消息中心、预警中心之间的职责边界是否继续压缩为更统一的个人事件收件箱模型。
