# 2026-04-11 客户关键字去重规则确认单

日期：2026-04-11

文档目标：把 `1.1` 客户关键字去重提醒，从“已有同名校验雏形”进一步压实到可进入设计和实现的规则颗粒度。

## 1. 当前锚点

当前仓内已确认以下现成能力：

1. 客户新建页已具备前端防抖式同名检查和即时提示。
2. 客户编辑页已具备修改名称时的重复校验。
3. `/api/customers` 创建接口已做后端“完全同名”重复检查，并返回冲突错误。
4. `/api/customers/[id]` 更新接口在名称变更时也会做后端“完全同名”重复检查。
5. 客户导入接口也已具备“客户名称已存在”的重复校验。

## 2. 当前问题

现状不是“没有去重”，而是“只有完全重名校验，还没有关键字相似提醒”：

1. 当前前后端规则都只覆盖 `customerName` 完全相等。
2. 当前提示文案是“客户名称已存在，请确认是否重复”，本质上是同名冲突，不是相似提醒。
3. 当前没有相似客户列表，不支持用户对候选客户做辅助判断。
4. 当前没有“继续提交”“打开已有客户详情”“仍然保存新客户”的分流动作设计。
5. 当前没有统一的关键词匹配规则，无法支撑“关键字去重提醒”这个需求表述。

## 3. 建议确认口径

### 3.1 匹配字段范围

建议第一版只使用以下 2 层匹配：

1. 主匹配字段：客户名称。
2. 辅助匹配字段：客户简称（若当前表未单独建字段，则先不启用）。

建议暂不把行业、联系人、地址等字段纳入相似度主规则，否则误报概率会明显上升。

### 3.2 相似规则分层

建议把去重规则分成两层，而不是一刀切：

1. 完全重复：标准化后名称完全相等。
2. 相似提醒：标准化后名称存在包含关系或高相似度文本匹配。

建议第一版的名称标准化规则至少包括：

1. 忽略首尾空格。
2. 统一大小写。
3. 去掉常见公司后缀差异影响，例如“有限公司”“股份有限公司”这类后缀应按统一规则归一。

### 3.3 触发时机

建议保持两段触发：

1. 输入中防抖检查：用于即时提示相似客户候选。
2. 提交前强校验：用于拦截完全重复客户。

这样可以兼顾体验和控制：

1. 用户在输入阶段就能看到提醒。
2. 真正重复的数据不会因为前端漏检而写入。

### 3.4 阻断策略

建议区分“完全重复”和“相似提醒”：

1. 完全重复：强阻断，不允许直接提交。
2. 相似提醒：弱提醒，允许用户确认后继续提交。

这比当前“只要重复就一律拦截”的单层规则更符合需求原意。

### 3.5 相似客户提示字段

建议相似客户列表至少展示：

1. 客户名称。
2. 所属地区。
3. 客户类型。
4. 联系人。
5. 创建时间或最近更新时间。

理由：

1. 这些字段足够辅助业务快速判断是否同一客户。
2. 不需要把全部客户详情塞进提示层。

### 3.6 页面动作

建议在提醒层提供以下动作：

1. 查看已有客户。
2. 继续创建新客户。
3. 取消提交并返回编辑。

当前只有错误提示，没有明确分流动作，后续应补齐。

### 3.7 接口职责

建议接口层职责统一为：

1. 创建 / 更新接口继续保留完全重复的硬校验。
2. 新增相似客户查询能力，用于前端即时提醒。
3. 不依赖前端已加载的客户列表作为唯一判断来源。

这意味着“关键字去重提醒”最终不能只做前端本地过滤，必须补后端统一规则。

## 4. 建议通过标准

若以下 7 项全部确认，则 ST-06 可视为规则验收通过：

1. 已确认第一版只围绕客户名称主规则展开。
2. 已确认完全重复与相似提醒是两层规则。
3. 已确认名称标准化口径。
4. 已确认输入中防抖检查 + 提交前硬校验的双时机。
5. 已确认完全重复强阻断、相似提醒可放行。
6. 已确认相似客户提示字段。
7. 已确认后端必须提供统一的相似客户查询能力。

## 5. 对开发的直接约束

本确认单通过后，后续实现应直接遵循以下约束：

1. 不能继续只依赖前端当前页客户列表做相似判断。
2. 不能把“关键字去重提醒”继续实现成“完全同名即报错”的旧逻辑翻版。
3. 创建和编辑场景都要共用同一套相似规则。
4. 提醒层必须提供查看已有客户和继续创建的分流动作。

## 6. 统一结论

客户去重这条线已经有了后端硬校验基础，因此并不缺“防重名”能力；真正缺的是“相似提醒”这层规则与交互。这个确认单的作用，就是把需求从“重名校验升级”收口成一套明确的关键字去重策略。
