# 数据大屏领导驾驶舱视觉实现规范

日期：2026-04-05

状态：待实施

## 1. 文档目的

本文用于为 `/data-screen` 的前端改造提供可直接执行的视觉与交互规范，目标不是单纯“做得更炫”，而是在保持经营驾驶舱业务价值的前提下，建立一套兼顾展示感、动态感和性能约束的实现标准。

本文重点解决四个问题：

1. 领导驾驶舱应该呈现什么样的视觉气质。
2. 哪些动态元素值得做，哪些不值得做。
3. 如何在首屏速度、长时间展示稳定性和视觉表现之间取得平衡。
4. 如何把这些要求落实到页面布局、组件样式、动效层级和前端实现方式。

## 2. 当前基础与限制

结合当前实现，数据大屏已经具备以下基础：

1. 页面已有较强科技感基调，当前主视觉依赖 `src/lib/tech-theme.ts`。
2. `src/app/data-screen/page.tsx` 已具备全屏、地图、排行、实时消息、模块切换等展示条件。
3. 当前组件大量使用 ECharts 和内联样式，技术上可以快速形成驾驶舱效果。

当前限制也很明显：

1. 页面主文件较大，状态较多，过多视觉增强会直接推高重渲染成本。
2. 地图、漏斗、排行、实时流本身已经是重模块，不能再叠加全局持续动画。
3. 当前主题偏“科技屏”但还不够“领导驾驶舱”，视觉层次和节奏感仍偏平。
4. 若不建立性能预算，后续很容易把页面做成演示时好看、正式使用时卡顿的系统。

## 3. 视觉定位

本轮视觉方向建议定义为：

高可信经营驾驶舱 + 克制的科技感 + 有节奏的动态反馈。

这意味着：

1. 页面要显得高级、稳定、专业，而不是赛博风装饰堆叠。
2. 视觉重点必须围绕经营信息层级，而不是背景特效。
3. 动效要体现“数据在流动”“状态在变化”“风险值得关注”，而不是全页持续发光闪烁。
4. 风格上延续当前深色科技基调，但应降低廉价霓虹感，提升管理驾驶舱气质。

## 4. 总体设计原则

### 4.1 信息优先

1. 首屏最先被看见的必须是经营摘要、风险和趋势，而不是背景和装饰。
2. 每个视觉决策都应回答一个问题：它是否帮助用户更快理解业务状态。

### 4.2 克制动效

1. 动效用于进入、切换、强调和反馈，不用于持续抢注意力。
2. 同屏同时出现的强动效数量必须严格受控。

### 4.3 分层渲染

1. 视觉层分为背景氛围层、结构容器层、数据内容层、交互反馈层。
2. 只允许数据内容层和交互反馈层参与核心首屏渲染。
3. 背景氛围层必须延后挂载或低频更新。

### 4.4 展示与运行兼容

1. 页面既要适合领导看屏，也要适合一线日常使用。
2. 全屏模式和普通模式应共享同一套视觉语言，但信息密度和留白允许不同。

## 5. 视觉风格规范

### 5.1 色彩方向

当前 `techTheme` 已有深色科技底座，本轮不建议推翻重做，而建议做“驾驶舱化收敛”：

1. 主背景继续采用深色体系，但减少纯黑大面积硬切，增加更细腻的深蓝灰层次。
2. 主强调色继续使用冷色高亮，但应把高饱和霓虹蓝控制在关键指标、选中状态和图表重点上。
3. 紫色只作为辅助渐变色使用，不再大面积占据主要视觉层。
4. 成功、警告、风险色必须承担明确业务语义，不能仅为装饰。
5. 背景发光和描边透明度应整体下调，避免页面长期观看刺眼。

建议的色彩分工：

1. 主色：冷蓝青色，用于关键高亮、选中、主曲线、重点数字。
2. 辅色：青绿或蓝紫，用于第二层图表和辅助状态。
3. 警告色：琥珀黄，用于临期、压力、目标缺口。
4. 风险色：红橙色，用于异常、下滑、滞留、高风险对象。
5. 中性色：深蓝灰、雾白，用于卡片、分割线、辅助文字。

### 5.2 字体与数字表现

1. 指标数字要比普通文本更有识别度，建议使用更紧凑、更有机械感的数字风格。
2. 标题层级应减少字号混乱，形成“页面标题 > 区块标题 > 卡片标题 > 辅助标签”的四级体系。
3. 大数字建议配合字重和间距控制，不要再单靠颜色制造强调。
4. 单位、同比、环比、标签必须做弱化处理，避免与主数值争抢注意力。

### 5.3 卡片语言

当前卡片已具备玻璃态和发光边框基础，下一步建议统一为：

1. 卡片分为主驾驶卡、分析卡、风险卡、列表卡四类。
2. 主驾驶卡强化边界、数字层级和顶部状态条。
3. 分析卡强调图表展示空间，减少无效边框和内阴影。
4. 风险卡强化状态色、优先级标识和操作入口。
5. 列表卡优先保证信息密度和扫描效率，不追求过度装饰。

## 6. 布局实现规范

### 6.1 页面结构

领导驾驶舱建议采用“三段式大框架 + 模块化嵌套”：

1. 顶部：品牌标题、时间、角色视图、全屏、刷新、筛选控件。
2. 中部：经营摘要、漏斗、区域热力、排行和专题诊断。
3. 底部或侧边：风险对象、实时流、待办动作、重点对象列表。

### 6.2 栅格原则

1. 页面以 12 栏或等效弹性网格组织，保证不同模块能稳定重排。
2. 关键经营摘要卡必须位于首屏折叠线以上。
3. 风险与动作区不能被挤压到视觉边缘，必须保留固定入口。
4. 地图与漏斗不可同时过小，否则只剩“有图”而没有信息价值。

### 6.3 全屏模式

1. 全屏不是简单放大，而应增加留白、提升区块节奏和观看距离友好度。
2. 全屏模式下，装饰层和光效可适度增强，但核心信息密度不能下降。
3. 全屏顶部控件应进一步收束，避免遮挡主体内容。

## 7. 动效实现规范

### 7.1 允许做的动效

建议重点做以下动效：

1. 页面首屏区块的分层入场。
2. 关键数字从旧值过渡到新值的滚动动画。
3. 图表切换维度时的短时转场。
4. 风险项出现或升级时的脉冲式高亮。
5. 抽屉、弹层、下钻详情的平滑展开与返回。

### 7.2 不建议做的动效

以下动效原则上不建议默认启用：

1. 全页持续漂浮粒子。
2. 所有卡片持续呼吸发光。
3. 所有图表默认长时循环播放动画。
4. 高频闪烁边框、无节制扫光、复杂 3D 旋转。
5. 与业务无关的装饰型大面积动效背景。

### 7.3 动效节奏

1. 入场动效应短而稳，控制在用户感知为“顺滑”而不是“等待”的区间。
2. 数字变化优先采用 300 到 800 毫秒级过渡。
3. 图表切换优先采用局部过渡，不触发整页动画。
4. 风险提醒应以低频强调为主，不应长时间持续闪动。

## 8. 性能预算与降级策略

### 8.1 性能预算

前端实现应建立明确预算意识：

1. 首屏先显示结构框架和核心摘要卡，不等待全部图表初始化完成。
2. 地图、排行、实时流、复杂图表采用懒加载或分块初始化。
3. 自动刷新只更新必要模块，不允许整页重新装配。
4. 单次筛选切换应尽量局限在相关区块重绘。
5. 长时间展示模式下，应避免持续高频定时器和全局动画循环。

### 8.2 降级模式

必须预留降级能力：

1. 对低性能终端关闭背景装饰层动画。
2. 对长时间大屏播放场景降低非核心动画频率。
3. 对数据量较大场景降低图表初始动画复杂度。
4. 对实时流和排行可采用节流刷新，而不是秒级强刷。

### 8.3 技术实现建议

1. 重模块优先使用懒加载和按需 hydration。
2. 背景氛围层建议使用 CSS 渐变、轻量遮罩和低成本动画，避免引入重 canvas 特效。
3. 数字动画使用轻量方案，不为简单数字翻滚引入高成本依赖。
4. 页面状态拆分后，应尽量减少大组件级别的重复 render。
5. ECharts 动画要按模块启用，不要默认所有图表全开。

## 9. 组件级实现建议

### 9.1 经营摘要卡

1. 强化数值、趋势箭头、环比同比、目标进度条。
2. 引入轻量数字滚动和顶部状态光带。
3. 卡片 hover 和 active 态要弱，不做后台管理页式按钮交互感。

### 9.2 漏斗与排行

1. 漏斗更强调转化率和阶段停留，而不是只显示柱形或面积。
2. 排行建议加入名次节奏和重点对象突出，但避免奖牌式过强娱乐化视觉。

### 9.3 区域热力图

1. 地图必须优先保证色阶清晰、区域可读、下钻反馈明确。
2. 地图动效只在切换维度、选中区域和下钻返回时出现。
3. 热力图配套侧边明细必须同步高亮，形成双通道理解。

### 9.4 风险与实时流

1. 风险项强调优先级、剩余时间、影响金额和下一步动作。
2. 实时流不追求信息滚动速度，而追求“有动静但不吵”。
3. 对新增事件可采用短暂高亮，而不是持续滚动跑马灯。

## 10. 实施顺序建议

视觉实现建议按以下顺序落地：

1. 先定义页面结构、卡片体系、色彩语义和全屏布局。
2. 再统一摘要卡、风险卡、图表卡和列表卡的基础样式。
3. 然后补关键动效，只做与信息反馈直接相关的部分。
4. 最后再加背景氛围层和展示增强项。

这个顺序可以避免项目一开始就陷入“先堆特效、后补结构”的错误路径。

## 11. 验收标准

视觉稿实现完成后，至少应满足：

1. 第一眼能明显感受到领导驾驶舱气质，而不是普通后台报表页。
2. 首屏能快速看到经营摘要、风险和重点对象。
3. 动态元素能够增强信息表达，而不是制造噪音。
4. 在普通模式和全屏模式下都保持稳定、清晰、专业。
5. 在真实数据和连续运行场景下，不出现明显卡顿、闪烁和过载。

## 12. 本文结论

领导驾驶舱完全可以做到“更好看、更多动态、但不慢”，关键不在于减少视觉追求，而在于把视觉增强纳入工程治理：

1. 核心信息先于装饰出现。
2. 动效只服务状态变化和用户注意力引导。
3. 背景氛围和装饰层必须受性能预算约束。
4. 所有展示增强都要为业务判断和现场展示服务。

在这个前提下，数据大屏既可以更受欢迎，也不会变成一个只适合截图、不适合真实运行的页面。
