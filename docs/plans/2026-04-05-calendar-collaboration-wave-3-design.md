# 工作台、日程、任务、消息中心模块演进计划第三波：协作日程能力升级

日期：2026-04-05

状态：已完成

## 1. 第三波目标

第三波聚焦“协作日程能力升级”，在不引入复杂排程引擎和审批流的前提下，把 `/calendar` 从个人记录页推进为可共享、可读边界明确、能表达重复规则和任务联动上下文的 canonical 日程页面。

本波目标为：

1. 保持 `/calendar` 作为唯一 canonical 日程页面。
2. 让日程从“个人记录”升级为“可邀请他人参与的协作事项”。
3. 把参与人配置从弱文本/隐含 JSON 能力升级为受控选择能力，并让参与人可读取共享日程。
4. 让协作日程和消息中心产生最小闭环：被邀请人收到消息提醒。
5. 让重复规则和任务联动不再只是底层字段，而是在 `/calendar` 中有受控展示与交互语义。

## 2. 变更分类

本波分类如下：

1. `requirement-gap`
2. `rule-conflict`

根因不是 `/calendar` 入口错误，而是日程模块虽然已经具备 `participants`、`reminder`、`repeat`、`relatedType/relatedId` 等结构字段，但没有形成受控 UI、消息联动和协作读写边界，导致“协作日程”只有数据壳，没有真实产品闭环。

## 3. 第三波定版结论

### 3.1 Canonical 页面与 API

1. `/calendar` 仍是唯一 canonical 日程页面。
2. `/api/schedules` 与 `/api/schedules/[id]` 仍是日程 domain API。
3. `/messages` 继续作为协作日程邀请和变更提醒的事件收件箱。

### 3.2 第三波范围边界

本波只做以下五件事：

1. 日程创建与编辑支持受控参与人选择。
2. 日程列表与详情向参与人开放读取，但修改和删除继续保持 owner-only。
3. 协作日程创建和关键更新向参与人发送消息提醒。
4. 日程创建与编辑支持受控重复规则配置，并在列表和详情中展示重复语义。
5. 任务自动生成的关联日程在 `/calendar` 中显式展示任务联动上下文。

本波明确不做：

1. 不实现复杂重复规则调度引擎。
2. 不引入日程审批流。
3. 不把日程直接并入任务主模型。

## 4. 实施要求

1. 参与人必须来自受控用户选择，不允许继续依赖随意文本拼接来形成协作成员。
2. 日程消息提醒必须以当前认证用户为发送边界，并进入消息中心统一收件箱。
3. `/api/schedules/[id]` 必须继续以当前用户 owner 作为修改和删除边界，不得因为协作能力而放松 owner 校验。
4. 共享日程的参与人只获得读取能力，UI 必须明确表现为只读，不得伪装成可编辑状态。
5. 重复规则本波只做受控配置、持久化和展示，不承担真正的重复展开调度职责。

## 5. 验证面

本波至少验证：

1. 日程 API 能正确归一化参与人并持久化。
2. 协作日程创建或关键更新时，会向参与人写入消息中心事件。
3. 共享日程的参与人可以读取列表和详情，但 `/api/schedules/[id]` 的 owner 修改边界仍然成立。
4. `/calendar` 页面能够创建、编辑并展示带参与人的协作日程。
5. `/calendar` 页面能够展示重复规则与任务联动标识，并对共享给当前用户的日程呈现只读状态。

## 6. 当前结论

1. `/calendar` 已支持受控参与人选择、提醒配置、重复规则配置，以及任务联动日程和共享只读日程的上下文展示。
2. `/api/schedules` 与 `/api/schedules/[id]` 已接入参与人归一化、重复规则归一化、协作消息写入，并把读取边界扩展到 owner 或 participant，同时继续保持修改/删除 owner-only。
3. focused API 回归 `tests/api/schedules/route.test.ts` 已通过 `4/4`，覆盖协作创建、owner-only 更新、参与人列表读取和参与人详情读取。
4. focused formal 回归 `tests/e2e/playwright/wave3-formal.spec.ts` 已在 `5004` 端口通过 `2/2`，验证了协作日程创建与消息可见、共享日程只读、重复规则展示，以及任务联动日程上下文可见。
5. 第三波现已整体完成；复杂重复执行引擎和更细粒度协作权限仍明确留在后续波次。

下一波可继续评估：

1. 重复规则与提醒调度的真实执行引擎。
2. 日程与任务的更强双向联动，而不只是消息通知。
3. 协作日程 owner 与参与人之间的更细粒度可见性和编辑权限。
