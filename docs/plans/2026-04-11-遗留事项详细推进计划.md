# 2026-04-11 遗留事项详细推进计划

## 1. 文档目标

本文件基于《遗留事项推进台账》，用于把当前仍未关闭的 `11` 类遗留事项转化为可执行、可验收、可回写的推进计划。

本计划不再重复解释历史背景，重点只解决四件事：

1. 当前推进范围是什么。
2. 当前推进边界是什么。
3. 每一类事项按什么顺序推进。
4. 每推进完一步，如何判定、回写和收口。

复用依据：

1. 《2026-04-11 遗留事项推进台账》
2. 《2026-04-11 待核需求项专项验收清单》
3. 《2026-04-10 Sheet1 Bug Fix Ledger》
4. 《2026-04-11 需求修改清单验收收敛结论》

## 2. 边界约束

### 2.1 范围边界

本轮推进范围严格限定为《遗留事项推进台账》中的 `11` 类事项，不新增、不替换、不扩写。

本轮明确纳入范围的只有：

1. `Sheet1` 的 `6` 类遗留 bug / 待确认问题。
2. `Sheet2` 的 `5` 类待核 / 待专项验收事项。

本轮明确不纳入范围的内容：

1. 已完成开发并通过研发侧验证的事项。
2. 已明确驳回、不纳入售前系统设计原则的事项。
3. 已闭环的第二梯队事项，包括周报 / 周报导出、工作台标签收拢、客户关键字去重提醒。
4. 任何未出现在当前遗留台账中的新增想法、体验优化建议或顺手改造项。

### 2.2 目标边界

本轮目标不是“一次性把所有遗留项全部开发完成”，而是把所有遗留项推进到明确出口。

允许的目标只有三种：

1. 复核后关闭。
2. 专项验收后关闭。
3. 明确转开发，并进入后续开发闭环。

本轮不以“大而全重构”“顺带统一整个模块”为目标，不允许借遗留项推进之名重新打开主干范围。

### 2.3 证据边界

任何遗留项若要从当前状态推进为“已关闭”或“转开发”，都必须具备最小证据。

最低证据要求如下：

1. 页面入口明确。
2. 验收账号或样本明确。
3. 最终结论明确。

对于高风险事项，至少补齐以下四类证据中的三类：

1. 页面现象。
2. 接口响应。
3. 数据库事实。
4. 原始台账事实。

没有证据，不允许直接改状态。

### 2.4 执行边界

执行时必须遵守以下约束：

1. `Sheet1` 先做复核与归因，再决定是否进入开发，不允许直接跳到改代码。
2. `Sheet2` 的待核项先跑专项验收，不允许在尚未验收前直接宣称“应该开发”。
3. 每个事项同一时刻只允许存在一个主状态，不允许并存“待核 + 开发中”双状态。
4. 每完成一项，必须即时回写台账，不允许累计到一批后再统一补录。
5. 若某事项需要业务口径确认，则在口径确认前不得启动正式开发拆解。

### 2.5 资源边界

本轮推进默认以最小必要资源组织，不新开额外大范围专项。

默认参与角色为：

1. 产品 / PMO：负责范围冻结、结论确认、台账回写。
2. 测试：负责复现、专项验收、证据沉淀。
3. 前端 / 后端：仅在事项被判定为代码问题或需要实现调整时介入。
4. 业务验收人：仅在涉及口径确认、权限边界、展示语义时参与判定。

## 3. 当前推进对象拆分

### 3.1 A 组：高风险 `Sheet1` 遗留项

1. `2.1` 项目权限异常。
2. `2.4` 重部署后名称丢失与跟进时间异常。
3. “当前跟进项目数和实际项目数量不匹配”。

这一组优先级最高，原因是直接影响权限安全、部署可信度和统计可信度。

### 3.2 B 组：`Sheet1` 业务口径遗留项

1. `1.1` 区域字段与区域口径治理。
2. `1.2` 客户历史中标金额取数口径。
3. “结算信息没有可填写内容，执行中阶段建议显示实施方案”。

这一组重点不在于先改页面，而在于先把字段语义、统计口径和产品归类讲清楚。

### 3.3 C 组：`Sheet2` 待核专项验收项

1. `3.1` 方案列表分页样式统一。
2. 附加项 A：新建项目页面跳动 / 页面自适应问题。
3. `2.5-2.7` 项目管理交互一致性。
4. `2.9-2.12` 投标字段与展示口径。
5. 附加项 D：合同管理新建页附件、签约模式、单位联动。

这一组重点是清空待核状态，不再让事项挂在“知道有问题，但没有最后结论”的中间层。

## 4. 推进阶段划分

### 4.1 第零阶段：边界冻结与样本确认

目标：在真正推进前，把范围和样本冻结下来。

执行动作：

1. 以《遗留事项推进台账》为唯一清单，确认本轮只推进 `11` 项。
2. 为每项指定页面入口、样本账号或数据样本。
3. 明确每项当前状态，只保留一个主状态。
4. 把本计划作为后续推进的唯一执行文档，不再另起平行台账。

阶段产出：

1. 冻结后的事项清单。
2. 每项对应的入口、样本、责任角色。

退出标准：

1. 所有 `11` 项都已具备推进起点。
2. 没有范围争议项和口径未冻结项悬置在阶段外。

### 4.2 第一阶段：A 组高风险问题复核

目标：先把权限、部署、统计三个高风险方向的真实性和归因拉实。

执行动作：

1. 对 `2.1` 进行目标账号复测，确认是否仍存在越权可见。
2. 对 `2.4` 组织一次重部署后样本复核，确认名称丢失和跟进时间异常是否可复现。
3. 对“当前跟进项目数和实际项目数量不匹配”补页面、接口、数据库三层核对。

阶段产出：

1. 每项形成“已排除 / 待修复 / 待进一步定位”结论。
2. 若确认是代码问题，补最小归因说明。

退出标准：

1. A 组三项都不再停留在“只有描述没有结论”的状态。
2. 至少形成关闭或转开发的单一出口。

### 4.3 第二阶段：B 组业务口径确认

目标：把 `Sheet1` 中剩余的字段和统计口径问题收口成可执行结论。

执行动作：

1. 对 `1.1` 明确区域唯一口径，核对页面字段、筛选选项、导出字段和存量数据。
2. 对 `1.2` 选取至少 1 组真实客户样本，核对“历史中标总额 / 历史最大中标金额”的来源与展示。
3. 对“结算信息 / 实施方案”条目先做归类判断，明确是缺陷、需求还是配置问题。

阶段产出：

1. `1.1` 的区域口径说明。
2. `1.2` 的统计口径说明与样本核对结果。
3. “结算信息 / 实施方案”条目的归类结论。

退出标准：

1. B 组三项均可落为“关闭”或“转开发”。
2. 不再保留“暂时如此”“以后再看”的模糊表述。

### 4.4 第三阶段：C 组待核项专项验收

目标：按现有专项验收清单，彻底清空 `Sheet2` 待核池。

执行动作：

1. 先验收 `2.5-2.7` 与 `2.9-2.12`，因为这两组与项目管理主链路耦合最高。
2. 再验收 `3.1` 方案分页样式统一。
3. 随后复核附加项 A 新建项目页面跳动。
4. 最后验收附加项 D 合同管理新建页三项能力。

阶段产出：

1. 每个编号形成“通过关闭”或“转开发”结论。
2. 对需转开发项补充目标口径、现状问题和建议改造方向。

退出标准：

1. `Sheet2` 本轮遗留项不再存在“待核”状态。
2. 所有待核项均已进入关闭或转开发池。

### 4.5 第四阶段：统一回写与管理收口

目标：确保推进结果同步沉淀到原始载体，避免后续再次出现多套口径。

执行动作：

1. 回写《遗留事项推进台账》状态。
2. 回写 `Sheet1` bug 台账或 `Sheet2` 需求收敛文档。
3. 对转开发项，补入后续开发池或专项开发计划。
4. 对关闭项，补关闭说明和证据位置。

阶段产出：

1. 更新后的统一台账。
2. 更新后的原始承载文档。

退出标准：

1. 不再存在“执行完了，但文档没改”的断层。
2. 管理口径与实际执行结果保持一致。

## 5. 分项执行要求

### 5.1 `Sheet1` 项执行要求

`Sheet1` 项统一按“复核优先”推进，不直接以开发为默认出口。

执行要求：

1. 每项先补样本，再补结论。
2. 若无法复现，必须记录复现条件、环境、样本，不允许只写“未复现”。
3. 若属于统计口径问题，必须写明取数来源、计数范围和展示位置。
4. 若属于权限问题，必须写明测试账号、角色和实际可见范围。

### 5.2 `Sheet2` 项执行要求

`Sheet2` 项统一按“专项验收优先”推进，不允许直接跳过验收进入开发。

执行要求：

1. 每项必须对应到专项验收脚本中的入口与检查点。
2. 验收结果只能是“通过关闭”或“转开发”。
3. 若转开发，必须补充最小改造目标，不允许只写“后续优化”。

## 6. 角色分工建议

| 角色 | 主要职责 | 交付物 |
| --- | --- | --- |
| 产品 / PMO | 范围冻结、口径确认、状态改写 | 结论确认、文档回写 |
| 测试 | 复现、专项验收、证据沉淀 | 样本记录、截图、结论 |
| 前端 | 页面行为、展示逻辑、交互问题定位 | 页面归因、改造建议 |
| 后端 | 接口、权限、统计口径、部署链路定位 | 接口归因、数据说明 |
| 业务验收人 | 权限边界、业务口径、展示语义确认 | 业务确认结论 |

约束说明：

1. 产品负责“收口”，不负责替代技术归因。
2. 测试负责“验证与证据”，不直接决定产品口径。
3. 开发只在事项被确认需要技术处理后进入主责，不提前卷入所有待核项。

## 7. 周期建议

### 7.1 第一个工作周期

聚焦 A 组与 B 组：

1. 完成高风险项复核。
2. 完成 `Sheet1` 口径项归类与说明。

目标：让 `Sheet1` 不再存在未入账、未归类、未闭环的游离事项。

### 7.2 第二个工作周期

聚焦 C 组：

1. 跑完 `Sheet2` 全部专项验收。
2. 把待核池清空。

目标：让 `Sheet2` 剩余项全部落入“关闭”或“转开发”。

### 7.3 第三个工作周期

聚焦回写与转开发项收口：

1. 对所有转开发项形成后续开发计划。
2. 对所有关闭项形成归档说明。

目标：把“遗留事项推进”从执行阶段切换到稳定维护状态。

## 8. 风险与控制措施

### 8.1 主要风险

1. 范围再次扩张，把已完成事项重新拉回。
2. 待核项在未验收前被直接判定为开发需求。
3. 只更新局部文档，导致总台账与实际状态脱节。
4. 业务口径未确认就先做实现，造成返工。

### 8.2 控制措施

1. 所有新增事项一律不进入本轮，统一挂到下一轮候选池。
2. 所有 `Sheet2` 待核项必须先走专项验收脚本。
3. 每项结论当日回写，不跨周期积压。
4. 涉及权限、统计、字段语义的问题，先做说明再做改造。

## 9. 回写规则

每项推进完成后，必须同步执行以下动作：

1. 更新《遗留事项推进台账》的当前状态、下一动作和结论。
2. 若属于 `Sheet1`，同步更新 bug 台账。
3. 若属于 `Sheet2`，同步更新需求收敛文档或专项验收清单。
4. 若转开发，则补入下一份开发计划，不允许只停留在“已转开发”的字面状态。

## 10. 完成判定

只有满足以下条件，才能判定本轮遗留推进计划完成：

1. `11` 项遗留事项全部离开当前悬空状态。
2. 不再存在“待核但无样本”“待复现但无环境”“已确认但未回写”的条目。
3. 所有关闭项都有依据，所有转开发项都有承接计划。
4. 原始文档、统一台账、实际状态三者一致。

## 11. 统一结论

本计划的核心不是扩大动作，而是压缩不确定性。

也就是说：

1. 先冻结边界，再推进事项。
2. 先补证据，再下结论。
3. 先清待核，再谈扩展。
4. 先回写台账，再宣称闭环。

后续只要严格按本计划执行，当前遗留项就不会继续停留在“知道有问题，但推进没有出口”的状态。
