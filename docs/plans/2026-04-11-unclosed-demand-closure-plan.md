# 2026-04-11 未闭环需求项闭环执行表

## 1. 目标

本文件用于对当前仍处于“部分落地”“待核”“转后续需求池”三类状态的需求项完成闭环管理。

这里的“闭环”不再停留在结论层，而要求每一项都进入以下三种明确出口之一：

1. 进入开发实施，并设置交付出口。
2. 进入专项验收，并设置判定出口。
3. 进入正式排期池，并设置转开发出口。

## 2. 闭环原则

1. 不再保留“悬空状态”。每一项都必须对应下一动作、责任角色和退出标准。
2. “部分落地”按开发闭环处理，不再继续保留为描述性状态。
3. “待核”按专项验收闭环处理，验收后只能转为“关闭”或“转开发”。
4. “转后续需求池”按正式排期闭环处理，不再只写“后续再说”。

## 4. 三类事项的具体闭环方式

### 4.1 部分落地项：直接转开发闭环

适用条目：`2.3`、`2.4`

执行要求：

1. 从“部分落地”改为“开发中 -> 已上线 -> 已验收”三段状态。
2. 不允许继续保留“局部做了但没有出口”的中间状态。
3. 验收必须包含页面行为、字段口径和最小现网回归证据。

闭环出口：

1. 功能上线。
2. 形成截图或运行态验证证据。
3. 在总台账中改写为“已关闭”。

### 4.2 待核项：直接转专项验收闭环

适用条目：`2.5-2.7`、`2.9-2.12`、`3.1`、附加项 A、附加项 D

执行要求：

1. 每项必须指定验收样本和页面入口。
2. 验收结果只能落到两种结论：
   - 已通过，关闭。
   - 未通过，转开发池。
3. 不再允许保留“待后面再看”的开放状态。

闭环出口：

1. 通过项进入关闭清单。
2. 未通过项进入下一轮需求池，并补具体改造目标。

### 4.3 转后续需求池项：直接转正式排期闭环

适用条目：`1.1`、`2.1`、`2.2`、附加项 B、附加项 C

执行要求：

1. 每项必须进入明确优先级池，不能只写“后续需求”。
2. 每项必须定义进入开发的触发条件。
3. 每项必须在排期文件中有承载位置。

闭环出口：

1. 已纳入排期池。
2. 已定义优先级和目标。
3. 后续进入开发后，再走开发闭环路径。
