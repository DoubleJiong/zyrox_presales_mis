# 售前管理系统权限矩阵设计稿

**目标：** 为售前管理系统建立一套统一、可实现、可审计、可测试的权限矩阵，支撑项目状态机、审批体系、认证统一和生产化治理。

**适用范围：** 用户登录、项目可见性、项目操作、审批动作、人员管理、方案管理、合同前置流程、仪表盘统计和管理类能力。

---

## 1. 设计背景

当前系统在权限方面至少存在以下问题：

1. 普通用户可以看到“无负责人项目”，造成敏感商机暴露风险。
2. 项目、方案、任务、仪表盘统计的权限口径并不一致。
3. 审批、阶段变更、对象编辑和查看之间没有统一权限框架。
4. 认证实现分裂，进一步放大了鉴权行为的不一致。
5. 一部分逻辑更像“接口各自判断”，而不是“系统统一授权”。

如果不先定权限矩阵，后续状态机、审批体系、认证统一和测试基线都会失去统一依据。

## 2. 设计目标

本设计稿希望实现以下目标：

1. 明确“谁可以看什么、改什么、审批什么、推进什么”。
2. 统一项目、审批、方案、人员、统计等核心域的权限口径。
3. 让权限判断可实现为稳定的 policy 层，而不是散落在 route 里。
4. 让权限边界可审计、可回归测试、可随阶段演进。
5. 为后续 RBAC 或 RBAC + ABAC 实现提供清晰输入。

## 3. 设计原则

### 3.1 默认拒绝原则

除公开页面和显式允许的健康检查类接口外，所有业务接口默认要求登录，并在未命中授权规则时拒绝访问。

### 3.2 先角色，再关系，再状态

权限判断建议按以下顺序进行：

1. 先判断用户属于什么角色。
2. 再判断用户与业务对象之间的关系。
3. 再判断业务对象当前所处阶段或审批状态。

也就是说，权限不是纯角色判断，也不是纯对象归属判断，而是三者组合判断。

### 3.3 查看与操作分离

能看到对象，不等于能修改对象；能修改对象，不等于能推进状态；能推进状态，不等于能审批。

### 3.4 审批权与发起权分离

审批发起人默认不应同时拥有对同一审批单的最终审批权，除非业务显式定义为自审场景。

### 3.5 敏感对象最小可见

无负责人项目、跨部门项目、商务材料、人员凭据、合同材料等敏感对象默认按最小权限开放。

### 3.6 权限必须可落审计

所有关键动作的权限判断结果都应能被记录，包括：

1. 谁发起了动作
2. 依据什么角色与关系授权
3. 对哪个对象执行了什么操作
4. 是否成功

## 4. 权限模型建议

建议采用“RBAC + 轻量对象关系约束 + 状态约束”的模型。

### 4.1 RBAC 层

负责定义用户的基础能力边界，例如：

1. 是否能管理人员
2. 是否能查看全部项目
3. 是否能审批某类单据
4. 是否能查看管理类统计

### 4.2 对象关系层

负责定义用户与具体对象的关系，例如：

1. 我是不是项目负责人
2. 我是不是交付负责人
3. 我是不是审批人
4. 我是不是创建人
5. 我是不是被指派成员

### 4.3 状态约束层

负责定义对象在某一阶段下，允许执行哪些动作，例如：

1. `opportunity` 阶段允许发起投标审批，但不允许进入执行
2. `bidding_pending` 阶段允许审批，不允许自由修改关键字段
3. `archived` 阶段只允许查询和有限补录

## 5. 角色定义建议

建议先收敛为以下七类角色。

### 5.1 系统管理员

负责平台级配置、用户与权限治理、特殊修复操作。

典型特征：

1. 不是所有业务审批的默认参与者
2. 但拥有系统维护和特批能力
3. 对敏感修复操作必须留审计日志

### 5.2 售前负责人

负责售前业务统筹、重点项目把关、投标和方案相关管理能力。

### 5.3 项目负责人

对具体项目的商机转项目、投标推进、协调和资料完整性负责。

### 5.4 交付负责人

对交付准备和执行阶段负责，不天然拥有售前立项和方案审批权限。

### 5.5 审批人

是审批链条中的角色集合，而不是一个孤立岗位。审批能力来自审批配置，不应默认赋给所有管理角色。

### 5.6 普通成员

参与项目执行、资料编辑、任务协作，但默认不具备敏感查看和审批权限。

### 5.7 财务 / 商务角色

负责报价、合同、商务确认等动作，默认不管理项目全生命周期。

## 6. 资源对象清单

建议权限矩阵至少覆盖以下对象：

1. 用户账号
2. 人员档案
3. 客户
4. 商机
5. 项目
6. 审批单
7. 方案
8. 合同材料
9. 仪表盘统计
10. 系统配置与审计日志

## 7. 动作类型清单

建议统一动作语义，至少包括：

1. `view`：查看
2. `list`：列表查看
3. `create`：创建
4. `edit`：编辑
5. `delete`：删除
6. `assign`：指派负责人或成员
7. `submit`：提交审批
8. `approve`：审批通过
9. `reject`：审批驳回
10. `withdraw`：撤回审批
11. `transition`：推动阶段流转
12. `export`：导出
13. `manage`：管理配置或权限

## 9. 条件允许规则

上表中的 `C` 必须进一步落到条件规则，否则会重新变成模糊权限。

### 9.1 项目查看权限

#### 系统管理员

可以查看全部项目，包括无负责人项目、已取消项目、归档项目。

#### 售前负责人

可以查看：

1. 全部售前阶段项目
2. 重点项目
3. 无负责人项目
4. 自己参与审批或监管的项目

#### 项目负责人

可以查看：

1. 自己负责的项目
2. 自己创建的项目
3. 自己参与审批流的项目

#### 交付负责人

可以查看：

1. 自己担任交付负责人的项目
2. 已进入 `delivery_preparing` 及之后阶段，且与自己有关联的项目

#### 普通成员

可以查看：

1. 自己被明确分配参与的项目
2. 自己创建的非敏感草稿对象

普通成员默认不能查看无负责人项目。

### 9.2 项目编辑权限

项目编辑应拆分为三类：

1. 基础信息编辑
2. 责任人编辑
3. 阶段变更

#### 基础信息编辑

允许：

1. 系统管理员
2. 售前负责人
3. 项目负责人（限本人负责项目）

#### 责任人编辑

允许：

1. 系统管理员
2. 售前负责人
3. 项目负责人仅可提名，不建议拥有最终变更权，除非业务确认

#### 阶段变更

阶段变更不应由普通编辑权限直接触发，必须走状态机服务，并满足前置条件。

### 9.3 审批权限

#### 发起审批

允许：

1. 系统管理员
2. 售前负责人
3. 项目负责人（限本人负责项目）
4. 财务/商务角色（限合同类审批）

#### 审批处理

允许：

1. 被配置为当前步骤审批人的用户
2. 系统管理员仅在特批或修复场景处理，且必须审计

默认规则：发起人不能审批自己发起的同一审批单。

### 9.4 方案权限

#### 查看

方案查看范围应跟随项目查看范围，不建议独立放大。

#### 编辑

允许：

1. 项目负责人
2. 被分配的方案成员
3. 售前负责人

#### 提交评审

允许：

1. 项目负责人
2. 售前负责人

### 9.5 合同材料权限

合同材料属于敏感对象，默认不向普通成员开放。

允许查看或编辑的人应至少满足以下之一：

1. 项目负责人
2. 交付负责人
3. 财务/商务角色
4. 当前合同审批链相关审批人
5. 系统管理员

### 9.6 仪表盘权限

仪表盘至少拆为两层：

1. 个人 / 本职责范围视图
2. 全局管理视图

默认规则：

1. 普通成员只能看自己范围
2. 项目负责人只能看与自己负责项目相关的统计
3. 售前负责人可看售前全局
4. 财务/商务角色可看商务口径全局，但不一定能看全部项目细节
5. 系统管理员可看系统级视图

## 10. 基于项目阶段的权限约束

即使角色满足，也必须受项目阶段约束。

| 项目阶段 | 允许重点动作 | 禁止重点动作 |
| --- | --- | --- |
| `opportunity` | 编辑基础信息、指派负责人、发起投标审批 | 直接进入交付、直接合同确认 |
| `bidding_pending` | 查看审批、撤回审批、审批处理 | 自由修改关键阶段字段 |
| `bidding` | 创建投标任务、方案编辑、报价准备、提交方案评审 | 直接归档 |
| `solution_review` | 查看评审、审批处理、驳回修订 | 绕过评审直接进入执行 |
| `contract_pending` | 上传合同材料、合同审批、商务修订 | 未经合同确认直接进入执行 |
| `delivery_preparing` | 指定交付负责人、准备交付计划 | 回退到任意早期阶段 |
| `delivering` | 执行协作、过程跟踪 | 发起售前立项类审批 |
| `settlement` | 结算与归档准备 | 回到投标或方案阶段 |
| `archived` | 查询、有限补录 | 普通业务编辑 |
| `cancelled` | 查询、特批修复 | 正常业务推进 |

## 11. 基于审批状态的权限约束

| 审批状态 | 发起人 | 当前审批人 | 其他相关人 | 系统管理员 |
| --- | --- | --- | --- | --- |
| `draft` | 可编辑、可提交、可取消 | N | N | 可查看 |
| `submitted` | 可查看、部分可撤回 | 可处理 | 可查看有限信息 | 可查看 |
| `in_progress` | 可查看 | 可处理 | 可查看有限信息 | 可查看 |
| `approved` | 可查看 | 只读 | 可查看 | 可查看 |
| `rejected` | 可查看、可重提 | 只读 | 可查看 | 可查看 |
| `withdrawn` | 可查看、可重提 | N | 可查看有限信息 | 可查看 |
| `cancelled` | 可查看 | N | 可查看有限信息 | 可查看 |
| `expired` | 可查看、可重提 | N | 可查看有限信息 | 可查看 |

## 12. 特批与例外机制

系统允许例外，但例外不能变成默认。

建议只允许两类特批：

1. 生产事故修复特批
2. 历史数据纠偏特批

特批要求：

1. 仅系统管理员可执行
2. 必须填写原因
3. 必须记录影响对象
4. 必须写入审计日志
5. 如影响主状态或审批状态，必须关联工单或决策记录

## 13. 技术落地建议

### 13.1 权限判断层次

建议把权限实现拆为三层：

1. `role policy`：角色基础能力
2. `resource policy`：对象关系与归属判断
3. `state policy`：阶段与审批状态约束

最终授权结果应是三层合取，而不是任意一层通过即可。

### 13.2 建议代码结构

```text
src/
  shared/
    auth/
    policy/
      roles.ts
      project-policy.ts
      approval-policy.ts
      solution-policy.ts
      dashboard-policy.ts
```

### 13.3 建议接口使用方式

所有 route handler 不直接写权限判断细节，而应调用统一 policy。

例如：

1. `canViewProject(user, project)`
2. `canEditProject(user, project)`
3. `canSubmitApproval(user, approvalRequest)`
4. `canApproveApproval(user, approvalRequest, step)`
5. `canViewDashboard(user, scope)`

## 14. 最低测试要求

### 14.1 权限单元测试

必须覆盖：

1. 无负责人项目对普通成员不可见
2. 项目负责人只能编辑自己负责项目
3. 非当前审批人不能审批
4. 发起人不能审批自己的审批单
5. 普通成员不能查看合同材料
6. 不同角色的仪表盘范围正确过滤

### 14.2 接口级权限测试

必须覆盖：

1. 未登录访问受保护接口返回 `401`
2. 已登录但无权限返回 `403`
3. 有权限时返回成功
4. 审批状态变化后权限也随之变化

### 14.3 审计测试

必须覆盖：

1. 特批动作产生审计日志
2. 审批动作产生审计日志
3. 敏感对象访问或修改可追踪

## 15. 待业务确认事项

以下问题需要业务最终拍板：

1. 项目负责人是否可以直接修改交付负责人，还是只能提名。
2. 财务/商务角色是否可以查看全部 `contract_pending` 项目，还是只看与自己相关项目。
3. 售前负责人是否拥有全部方案查看权，还是仅限售前范围。
4. 无负责人项目是否允许存在“待认领池”，若允许，哪些角色可见。
5. 已归档项目的补录权限开放到什么粒度。

## 16. 结论

权限矩阵不是附属规则，而是整个系统治理的中轴之一。

它直接决定：

1. 哪些对象是安全的
2. 哪些流程是可信的
3. 哪些审批是真正有效的
4. 哪些统计口径是可解释的
5. 后续认证统一和测试体系如何落地

因此，建议把这份权限矩阵设计稿作为下一批治理定版资产之一，并在后续继续拆分为：

1. 角色字典
2. 资源权限表
3. 阶段约束表
4. 审批权限表
5. 权限测试清单
