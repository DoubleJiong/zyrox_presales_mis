# 2026-04-11 周报规则确认单

日期：2026-04-11

文档目标：把第二梯队中“周报 / 周报导出”的业务规则，从首轮基线进一步压实到可直接进入设计和开发的颗粒度。

## 1. 当前锚点

当前仓内已确认以下现成能力：

1. 工作台页已提供“一键生成报告”入口，可打开周报、月报、年报生成对话框。
2. `WeeklyReportDialog` 已具备生成、预览、编辑、复制、下载交互。
3. `/api/reports/weekly`、`/api/reports/weekly/[id]` 以及 `bus_weekly_report` 表结构已存在。
4. 报表中心组件已接入周报列表、详情、删除等链路。
5. 现有报表展示层已经有 `personal / global` 两种周报类型文案锚点。

## 2. 当前问题

现状不是“没有周报”，而是“已有雏形，但规则和写入闭环未统一”：

1. 工作台生成链路当前以个人周报为主，没有明确的管理视角入口。
2. 工作台弹层中的“保存”调用 `POST /api/reports/weekly`，但当前接口实现是重新生成内容，不是按编辑结果落正式周报记录。
3. 当前“下载”是浏览器本地导出 `.txt`，不是受控导出能力。
4. 详情接口和列表接口已存在，但工作台临时生成链路与周报中心的正式记录链路没有真正统一。
5. 权限边界目前没有清晰定稿，尤其是他人周报查看和导出权限。

## 3. 建议确认口径

### 3.1 周报视角范围

建议先按两层视角收口：

1. 个人周报：当前登录用户自己的周工作总结。
2. 管理周报：管理角色按组织或人员范围查看下属周报列表。

暂不建议本轮直接引入更多自由组合视角，否则导出字段、权限和页面入口都会同时膨胀。

### 3.2 正式数据落点

建议明确 `bus_weekly_report` 为唯一正式落点。

对应规则：

1. 工作台弹层中的“生成”只负责生成预览内容。
2. 工作台弹层中的“保存”必须写入 `bus_weekly_report`。
3. 报表中心读取、查看、删除、后续导出都以正式周报记录为准。
4. 不再保留“工作台临时文本”和“报表中心正式记录”两条平行语义。

### 3.3 周期规则

建议本轮保持以下口径：

1. 周报按自然周统计，周一至周日为一个完整周期。
2. 周报主键唯一性沿用当前表约束：`type + userId + weekStart`。
3. 跨月周不拆成两份周报，仍以周为单位存储和导出。
4. 当周无数据时允许生成空态周报，但需明确提示“本周暂无有效工作记录”。

### 3.4 统计字段范围

建议第一版周报正文与结构化字段统一覆盖以下内容：

1. 周期信息：年份、周次、周起止日期。
2. 基础统计：工作天数、累计工时、完成任务数、进行中任务数。
3. 工作日志摘要：日期、工时、工作类型、工作内容摘要。
4. 已完成任务：任务名称、所属项目、完成日期。
5. 进行中任务：任务名称、所属项目、进度、截止日期。
6. 生成正文：面向复制 / 导出的自然语言周报内容。

建议暂不在第一版加入自由自定义模块，否则会把规则确认扩大到模板引擎层。

### 3.5 导出格式

建议分两层处理：

1. 第一版正式导出格式：`txt`。
2. 结构化导出预留：Excel 或 CSV 作为后续增强项。

原因：

1. 当前页面已有 `.txt` 预览与导出心智，改造成本最低。
2. 若本轮直接承诺 Excel，则必须同时定义列头、工作日志明细展开方式和多表结构，范围会明显变大。

### 3.6 文件命名规则

建议统一为：

1. 个人周报：`周报_姓名_2026年第15周.txt`
2. 管理视角导出单人周报：`周报_姓名_2026年第15周.txt`
3. 管理视角导出汇总包：后续增强，不纳入本轮。

### 3.7 权限边界

建议本轮先收口为：

1. 普通用户：只能生成、保存、查看、导出自己的个人周报。
2. 管理角色：可查看权限范围内人员的周报列表与详情。
3. 管理视角导出：仅对具备相应报表权限的管理角色开放。
4. 暂不允许任意用户查看他人周报详情。

### 3.8 页面入口

建议分两类入口，不再混淆：

1. 工作台：承担“快速生成、即时预览、保存当前周报”的快捷入口。
2. 报表中心：承担“查看历史周报、管理周报记录、按权限查看他人周报”的正式入口。

## 4. 建议通过标准

若以下 8 项全部确认，则 ST-01 可视为规则验收通过：

1. 已确认周报只分个人视角与管理视角。
2. 已确认 `bus_weekly_report` 为唯一正式落点。
3. 已确认自然周口径与跨月周规则。
4. 已确认无数据周的空态处理。
5. 已确认第一版结构化字段范围。
6. 已确认第一版正式导出格式为 `txt`。
7. 已确认个人 / 管理角色的查看与导出权限边界。
8. 已确认工作台与报表中心的入口职责分工。

## 5. 对开发的直接约束

本确认单通过后，后续实现应直接遵循以下约束：

1. `POST /api/reports/weekly` 不能再只做“重新生成”，而必须支持正式保存语义。
2. 工作台弹层不能继续把本地下载当成正式导出闭环。
3. 报表中心与工作台必须读写同一份正式周报记录。
4. 权限控制必须从接口层落地，而不只停留在页面入口隐藏。

## 6. 统一结论

周报这条线现在已经不缺“有没有能力”，而是缺“正式能力边界”。本确认单的目标就是把这条线从“临时生成工具”收口成“正式周报能力”。
