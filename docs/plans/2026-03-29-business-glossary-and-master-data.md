# 售前管理系统统一术语表与主数据边界

**目标：** 为第二阶段提供唯一业务术语来源，结束同一对象多套叫法、多套表义和多套判断口径并存的问题。

---

## 1. 术语治理原则

1. 中文用于业务沟通，英文标准名用于代码、接口、测试和审计。
2. 一个业务对象只能有一个未来主资源名。
3. 过渡期允许兼容字段或兼容表存在，但必须标注“兼容层”身份。
4. 术语表优先级高于历史代码习惯。

## 2. 核心对象术语表

| 中文术语 | 标准英文名 | 当前承载位置 | 第二阶段结论 |
| --- | --- | --- | --- |
| 登录账号 | `user_account` | `sys_user` 中登录字段 | 是认证主体，不等于完整人员档案 |
| 人员档案 | `staff_profile` | 当前仍主要落在 `sys_user` 与人员页面 | 现阶段与账号仍混存，第三阶段再拆生命周期 |
| 线索 | `lead` | `bus_lead` | 表示最早期商业线索，不等于项目 |
| 客户 | `customer` | `bus_customer` | 客户主数据，允许被项目引用，不应靠冗余名称作为主来源 |
| 项目 | `project` | `bus_project` | 核心业务主对象，承载售前到交付前后主链路 |
| 项目主阶段 | `project_stage` | `projects.projectStage` | 第二阶段认定为业务主状态字段 |
| 项目兼容状态 | `project_status` | `projects.status` | 视为兼容/技术字段，后续逐步退化 |
| 审批单 | `approval_request` | 当前未形成统一表，散落在若干业务表/route | 第二阶段认定为必须独立建模的一等实体 |
| 方案主模型 | `solution` | `bus_solution` 及相关子表 | 第二阶段认定为未来唯一主模型 |
| 历史方案模型 | `scheme` | `bus_scheme` 及相关表 | 第二阶段认定为历史兼容模型，不再作为未来主资源名 |
| 仪表盘 | `dashboard` | `api/dashboard`、查询服务 | 统计展示层，不得反向定义业务口径 |

## 3. 项目相关术语定版

### 3.1 `projectStage`

作为项目业务主阶段字段，承载项目在主链路中的事实位置。

第二阶段结论：

1. 所有阶段流转设计以 `projectStage` 为准。
2. 状态机、审批前置条件、统计口径、权限判断优先引用 `projectStage`。
3. 后续不允许新增以 `status` 作为主业务阶段判断的实现。

### 3.2 `status`

当前项目表仍存在 `status` 字段，并被部分查询和统计使用。

第二阶段结论：

1. `status` 暂视为兼容字段。
2. 允许短期保留用于旧页面、旧筛选或技术兼容。
3. 第三阶段开始，必须逐步把主业务语义从 `status` 迁出。

## 4. 审批相关术语定版

1. 审批单标准名为 `approval_request`。
2. 审批步骤标准名为 `approval_step`。
3. 审批事件标准名为 `approval_event`。
4. “提交审批”“审批通过”“审批驳回”“撤回审批”必须视为审批实体事件，而不是项目字段临时改写。

## 5. 方案相关术语定版

1. `solution` 为未来唯一主资源名。
2. `scheme` 视为历史遗留实现，不再扩展新主能力。
3. `solution_sub_scheme` 仅表示 `solution` 体系内的子方案结构，不构成恢复 `scheme` 主模型地位的依据。

## 6. 组织与身份术语定版

1. `user_account` 表示登录与认证主体。
2. `staff_profile` 表示人员资料与组织属性。
3. 现阶段两者仍物理混存于 `sys_user`，但概念上必须区分。
4. “谁能登录”“谁能管理人员”“谁能审批”“谁对项目负责”属于不同维度，不允许再混写为单一账号属性。

## 7. 第二阶段直接禁止的术语漂移

1. 不再把 `project.status` 和 `project.projectStage` 视为同义字段。
2. 不再把 `scheme` 和 `solution` 同时作为未来主模型。
3. 不再把“审批动作”和“阶段变化”视为同一件事。
4. 不再让 dashboard 指标定义反向成为业务规则来源。

## 8. 对第三阶段的约束

第三阶段所有重构、迁移、policy、测试命名必须引用本术语表。若实现需要新增术语，必须先补本文档，再允许进代码。
