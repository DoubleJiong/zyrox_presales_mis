## 2. 二期目标

本期不是在现有页面上继续补卡片，而是完成“一个数据大屏、两个并行主视角、一个统一下钻层”的产品重构。

本期结束后，至少要达到以下结果：

1. 导航层只保留一个“数据大屏”入口，不再把团队执行相关能力做成平行模块。
2. 页面层形成并行的“区域视角”和“人员视角”，二者是一级视角，不是上下级钻取关系。
3. 区域视角能够稳定呈现全国与浙江两类地图下的核心经营指标，不再出现浙江全零占位误导。
4. 人员视角能够稳定下钻到“人”与“事项”，回答“谁在忙、忙什么、哪里异常、具体是哪件事”。
5. 区域与人员两类视角共用统一筛选模型、统一钻取层、统一指标口径和统一权限边界。
6. 二期上线范围具备 focused API 测试、页面级回归和业务验收脚本。

## 3. 范围边界

### 本期要做

1. 收口信息架构、导航入口和页面骨架。
2. 修复浙江地图数据链路，补齐浙江维度核心指标最小可用集。
3. 重构区域视角页面和其读模型、图表、联动逻辑。
4. 新建人员视角页面和其读模型、图表、事项穿透能力。
5. 建立统一下钻层，承接区域、人员、任务、项目、客户、方案等对象明细。
6. 建立二期专项测试、验收和发布检查清单。

### 本期不做

1. 不在本期内重做整套绩效管理模型。
2. 不把绩效规则配置、考核流程引入本期主链路。
3. 不为浙江地图做全省所有口径一次性全量精细化建模，先完成 P0 指标真实可用。
4. 不新开第二个“团队驾驶舱”导航入口。

## 4. 拆解原则

1. 先修真实数据链路，再做视觉表达，不允许继续用占位零值掩盖数据缺口。
2. 先收敛统一筛选和读模型边界，再铺页面组件，避免再次形成页面直拉接口的并行旁路。
3. 区域视角与人员视角并行推进，但浙江专项必须前置，否则区域视角验收会失真。
4. 所有页面能力必须能落到“指标解释、对象明细、业务动作”三层之一，避免新增展示噪音。

## 5. 研发工作流与里程碑

建议按四个阶段推进：

1. `M1 基座收口`：完成导航收口、统一筛选模型、浙江链路止血。
2. `M2 区域视角可用`：完成区域视角首屏、地图联动、区域下钻。
3. `M3 人员视角可用`：完成人员概览、负载风险、事项穿透。
4. `M4 统一验收发布`：完成统一下钻层、回归基线、业务验收与上线准备。

建议最小可上线范围为：

1. DS2-01 至 DS2-05
2. DS2-07 至 DS2-10
3. DS2-12
4. DS2-14

## 7. 任务明细

## DS2-01 导航与信息架构收口

**任务目标：** 把“数据大屏”和“团队驾驶舱”收敛为一个统一入口，并建立二期页面骨架。

**涉及范围（预计）：**

- `src/components/navigation-menu.tsx`
- `src/components/mobile-nav.tsx`
- `src/app/data-screen/page.tsx`
- 数据大屏二期视图切换相关组件

**执行要求：**

1. 导航中仅保留“数据大屏”统一入口。
2. 页面内部形成一级视图切换，不再通过独立路由表达团队执行模块。
3. 页面骨架明确区域视角、人员视角、专题视角的切换位。

**完成定义：**

1. 导航结构对桌面端和移动端口径一致。
2. 页面刷新、分享链接、无权限访问均有明确行为。
3. 不破坏现有 `/data-screen` 的 canonical 路由地位。

## DS2-02 统一筛选模型与 URL 协议

**任务目标：** 为区域视角和人员视角建立共享的筛选模型、查询协议和参数校验层。

**执行要求：**

1. 统一视角、时间范围、区域层级、异常筛选、对象关键词等公共参数。
2. 明确不同视角专属参数，例如地图层级、人员分组、事项状态。
3. 统一前端状态、URL 参数和 API 查询结构映射。
4. 对非法参数和缺省参数建立兜底行为。

**完成定义：**

1. 切换视角时公共筛选保留，专属筛选按规则重置。
2. 刷新页面后主筛选状态可恢复。
3. 不再出现页面组件各自维护一套筛选状态的情况。

## DS2-03 浙江地图数据链路止血

**任务目标：** 先解决浙江地图“看起来有功能，实际上是零值占位”的问题，恢复区域视角的可信度。

**涉及范围（预计）：**

- `src/lib/data-screen-map.ts`
- `src/app/api/data-screen/heatmap/route.ts`
- 浙江地图映射和排行相关组件

**执行要求：**

1. 修复前端在浙江模式下强制返回零值地图数据的问题。
2. 补齐后端浙江热力图最小可用指标，至少覆盖客户数、项目数、活跃项目数、合同金额或项目金额之一。
3. 对暂未具备真实来源的指标明确标注“未建设”或“暂不展示”，不能静默填零。
4. 输出浙江专项问题清单，区分“代码问题”和“源数据问题”。

**完成定义：**

1. 浙江地图不再全量呈现默认零值。
2. 浙江区域排行与地图颜色能反映真实数据差异。
3. 业务可区分“没有数据”和“该指标暂未建设”。

## DS2-04 区域视角初始化读模型

**任务目标：** 建立区域视角首屏初始化接口，承接首屏指标、地图主数据和首批排行数据。

**执行要求：**

1. 输出区域总览 KPI、地图主数据、区域排行、趋势摘要的统一响应结构。
2. 统一全国与浙江两种地图模式的响应协议。
3. 接入角色权限、对象范围和缓存策略。
4. 为后续区域下钻保留 drilldown token 或标准对象键。

**完成定义：**

1. 区域视角首屏可由单次初始化请求完成主要渲染。
2. 不要求前端二次拼装多个接口才能得出主指标。
3. 空数据、越权、异常参数均有稳定返回结构。

## DS2-05 区域视角页面首屏重构

**任务目标：** 以地图为主舞台，完成区域视角首屏和图表联动区重构。

**执行要求：**

1. 按线框蓝图落位顶部核心指标区、中心地图区、左右联动分析区、底部多口径区。
2. 复用现有成熟图表组件，避免重复造轮子。
3. 建立地图 hover、点击、排行切换、图表切换之间的联动关系。
4. 状态上明确加载态、空态、异常态和“口径未建设”态。

**完成定义：**

1. 用户能从区域视角直接回答“哪里强、哪里弱、为什么异常”。
2. 全国与浙江模式切换行为稳定，不闪烁、不出现口径错位。
3. 页面主结构不再依赖多个历史模块的拼贴式组合。

## DS2-06 区域视角下钻层与排行联动

**任务目标：** 让区域视角不仅能看图，还能落到区域对象明细。

**执行要求：**

1. 建立区域下钻侧板或抽屉，承接区域 KPI、客户、项目、风险、人员协同摘要。
2. 建立地图选择、排行点击、异常榜单点击的统一联动协议。
3. 区域下钻层支持继续跳转到项目、客户等 canonical 页面。

**完成定义：**

1. 区域内异常对象可追溯，不需要离开数据大屏再手工检索。
2. 同一区域对象在地图、榜单、图表三处的口径一致。

## DS2-07 人员视角初始化读模型

**任务目标：** 建立人员视角的统一读模型，支撑团队规模、负载风险、事项明细和角色分组统计。

**执行要求：**

1. 输出团队总人数、活跃人数、过载人数、低活跃人数、风险人数等汇总指标。
2. 输出按角色组、区域、项目参与、事项状态聚合的统计结构。
3. 输出人员列表与事项列表的标准结构，支持分页或分段加载。
4. 明确事项对象的最小字段集，包括负责人、事项类型、优先级、状态、截止时间、所属项目或客户。

**完成定义：**

1. 人员视角主信息可由统一接口或少量标准接口稳定返回。
2. 人员与事项数据不再散落在多个历史模块接口里临时拼装。

## DS2-08 人员视角页面首屏重构

**任务目标：** 完成人员视角的统计层、结构层和图表层展示。

**执行要求：**

1. 按线框蓝图落位团队概览、角色负载、风险分布、事项趋势、人员列表等区域。
2. 强化“谁异常”“谁过载”“谁低活跃”的一眼识别能力。
3. 与区域视角保持相同的控制条、切换语义和状态反馈模式。
4. 预留后续绩效统计并入的展示插槽，但不在本期内实现绩效模块重构。

**完成定义：**

1. 用户能在人员视角快速识别重点跟进人员。
2. 页面不是单纯名单页，而是具有统计分析和异常识别能力。

## DS2-09 人员事项穿透与异常筛选

**任务目标：** 把人员视角真正落到“人”和“事项”。

**执行要求：**

1. 支持从汇总卡、风险图、角色组图表点击进入人员范围。
2. 支持从人员列表继续下钻到其事项列表。
3. 支持事项级异常筛选，例如逾期、高优先级未推进、长时间未更新、跨项目过载。
4. 支持从事项跳转到项目、任务、客户、方案等 canonical 页面。

**完成定义：**

1. 可以明确看到每个人当前在做什么。
2. 可以按异常类型快速定位需要管理动作的事项。

## DS2-10 统一下钻层与业务跳转

**任务目标：** 建立统一的详情层和跳转协议，避免区域和人员各做一套抽屉体系。

**执行要求：**

1. 定义统一 drilldown model，至少覆盖区域、人员、事项、项目、客户、方案对象。
2. 建立统一详情层容器，支持不同对象类型挂载不同内容片段。
3. 建立统一跳转入口和返回逻辑。

**完成定义：**

1. 用户在任意视角点击对象都走统一交互模式。
2. 后续专题视角可以直接复用该下钻层。

## DS2-11 专题视角与扩展插槽

**任务目标：** 为跨视角专题分析保留可落地扩展位，避免二期完成后又重新拆页面。

**执行要求：**

1. 建立专题视角页签或入口插槽。
2. 首期可只落 1 个轻量专题，例如风险专题或方案支撑专题。
3. 与统一筛选和统一下钻层保持一致。

**完成定义：**

1. 专题视角具备扩展能力，但不影响 P0 主链路上线。

## DS2-12 图表组件收敛与视觉统一

**任务目标：** 利用现有图表资产完成统一表达，避免新旧模块视觉语言割裂。

**执行要求：**

1. 盘点并复用已有图表组件，例如趋势图、饼图、仪表、雷达、排行等。
2. 收敛二期页面中的标题区、图例、空态、异常态、加载态规范。
3. 对地图主舞台、人员统计区、底部趋势带建立统一间距和信息层级。
4. 控制动画与视觉装饰，不影响长时间运行稳定性。

**完成定义：**

1. 区域和人员两个视角看起来属于同一套驾驶舱系统。
2. 页面信息密度提升，但不会因为状态不统一造成理解负担。

## DS2-13 权限边界与性能兜底

**任务目标：** 避免二期重构过程中再次出现权限口径分裂和页面过重问题。

**执行要求：**

1. 复核前端权限显隐与后端权限校验是否一致。
2. 为首屏初始化接口、下钻接口和榜单接口定义缓存与失效策略。
3. 对重图表、明细表和非首屏模块采用懒加载或分层加载。
4. 建立长时运行和频繁切换下的性能观察点。

**完成定义：**

1. 不再出现前端可见但后端 403 的同类问题。
2. 首屏可控，切换和下钻不会造成明显卡顿。

## DS2-14 自动化测试与业务验收

**任务目标：** 为二期建立最小可依赖的工程门禁和业务验收脚本。

**执行要求：**

1. 新增 focused API 测试覆盖区域视角初始化、浙江热力图、人员视角初始化、统一下钻层。
2. 新增 focused Playwright 场景覆盖视角切换、浙江数据可见、人员到事项穿透、详情层跳转。
3. 输出业务验收脚本，分别覆盖区域管理者和团队管理者视角。
4. 输出指标对账模板和问题台账。

**完成定义：**

1. 二期主链路具备基本自动化门禁。
2. 业务能够按脚本完成陪跑和签字。

## DS2-15 联调收口与发布准备

**任务目标：** 在代码完成后，把联调、验收、发布动作收成闭环。

**执行要求：**

1. 输出联调问题台账、遗留项清单和发布建议。
2. 明确正式环境验证路径、账号范围和截图证据要求。
3. 固化上线前检查项，包括权限、指标、性能、回归和回滚准备。

**完成定义：**

1. 二期不是“开发完成即结束”，而是具备上线前闭环材料。

## 10. 风险与前置提醒

1. 浙江问题不能只看页面表现，必须拆成“代码修复”和“底层数据补齐”两条线分别跟踪。
2. 人员视角若没有统一事项对象模型，后续一定会退化为多个模块明细拼接，建议尽早冻结最小字段集。
3. 若统一下钻层后置过晚，区域和人员两条线会各自长出一套详情交互，后续收口成本会明显上升。
4. 绩效统计并入大屏是后续方向，本期只预留插槽，不应影响当前 P0 范围交付。

## 11. 验收口径

二期验收建议按以下四类问题判断是否通过：

1. 信息架构是否收口：是否只剩一个“数据大屏”统一入口。
2. 区域视角是否可信：全国、浙江是否都能展示真实可解释数据。
3. 人员视角是否可管理：是否能落实到每个人及其具体事项。
4. 工程基线是否闭环：是否具备 API 回归、页面回归、业务验收和上线检查材料。
