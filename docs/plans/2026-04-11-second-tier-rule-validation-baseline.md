# 2026-04-11 第二梯队规则验收基线

日期：2026-04-11

文档目标：基于当前代码与页面落点，形成第二梯队三项需求的首轮规则验收基线，明确哪些事项已经有现成实现锚点，哪些事项仍缺业务定义。

## 1. 验收方式

本轮规则验收不依赖口头回忆，而是直接以当前仓内页面、组件、接口和数据表为准。

本轮输出只回答三个问题：

1. 当前系统有没有现成落点。
2. 当前系统离目标需求还差哪一层。
3. 哪些项可以直接进入开发设计，哪些项必须先补业务定义。

## 2. 范围

本轮仅覆盖第二梯队三项：

1. 附加项 B：周报 / 周报导出。
2. 附加项 C：工作台标签收拢到工作台下方。
3. `1.1`：客户关键字去重提醒。

## 3. 首轮基线结论

## 4. 分项结论

### 4.1 附加项 B：周报 / 周报导出

#### 4.1.1 已确认的代码锚点

1. 工作台页面已提供“一键生成报告”入口，且直接暴露周报、月报、年报按钮。
2. 周报弹层组件已存在，具备预览、编辑、复制、下载交互。
3. 周报生成接口已存在，周报明细接口和 `bus_weekly_report` 表结构也已存在。
4. 报表中心组件也已接入周报列表与详情读取链路。

#### 4.1.2 当前能力现状

1. 当前工作台入口的周报生成以“个人视角”周报为主，而非部门 / 团队周报。
2. 弹层中的“下载”目前是前端本地生成 `.txt` 文件，并不是受控的导出链路。
3. `POST /api/reports/weekly` 当前会重新生成周报内容，但并未按页面提交内容写入 `bus_weekly_report`。
4. 仓内已经有周报详情路由和数据表，但工作台入口与持久化链路没有真正接上。

#### 4.1.3 规则缺口

1. 周报是否只做个人周报，还是包含管理视角、团队视角，目前未定。
2. 导出目标格式未定：纯文本、Excel 还是其他格式。
3. 导出字段口径未定，当前页面预览内容与结构化导出列头未建立映射。
4. 查看权限、导出权限、他人周报访问权限边界未定。

#### 4.1.4 当前结论

该项不是从零开始，已经具备真实页面、真实接口、真实表结构锚点，因此可以直接进入设计细化。

但在进入编码前，必须先确认四件事：

1. 周报范围是个人还是多视角。
2. 导出格式和列头口径。
3. 保存链路是否以 `bus_weekly_report` 为唯一正式落点。
4. 权限边界如何划分。

### 4.2 附加项 C：工作台标签收拢到工作台下方

#### 4.2.1 已确认的代码锚点

1. 当前工作台页面已存在“工作台模块导航”卡片，负责统一汇总工作台相关模块入口。
2. 当前工作台还存在今日优先队列、今日待办、本周日程、重点项目、个人事件收件箱等区域。
3. 经过首轮检索，尚未在当前工作台页面上确认到独立命名的“标签区”组件或标签数据结构承载。

#### 4.2.2 当前能力现状

1. 现有页面更像“模块导航 + 信息卡片 + 动态收件箱”的组合，不是已经成型的“标签区迁移”问题。
2. 因为目标对象不够明确，现阶段无法精确定义“收拢前位置”“收拢后位置”“哪些标签需要迁移”。

#### 4.2.3 规则缺口

1. 本需求中的“标签”具体指模块导航入口、筛选标签、业务状态标签，还是另一个未收敛的入口区。
2. “工作台下方”具体是摘要卡片下方、模块导航下方，还是整个首屏内容区下方。
3. 标签数量较多时是折行、横向滚动还是折叠。
4. 标签点击后仍跳原页面，还是改成聚合视图，目前未定。

#### 4.2.4 当前结论

该项当前不具备直接开发条件。

先要补的不是代码任务，而是业务对象定义：

1. 先指认“要迁移的标签到底是哪一块”。
2. 再定义落点、层级和响应式规则。
3. 业务口径确认后，才能进入 ST-04、ST-05 的开发拆解。

### 4.3 `1.1` 客户关键字去重提醒

#### 4.3.1 已确认的代码锚点

1. 客户新建弹窗已存在客户名称即时检查逻辑。
2. 当前页面使用前端防抖方式，在本地 `customers` 列表中检查是否存在同名客户。
3. 创建和编辑都已经有“客户名称已存在，请确认是否重复”的阻断式提示。

#### 4.3.2 当前能力现状

1. 当前逻辑只校验 `customerName` 完全相等。
2. 前端即时提示基于页面已加载客户列表，但创建、编辑、导入接口都已经有后端“完全同名”重复校验。
3. 当前页面提示的是“同名客户已存在”，不是“关键字相似客户提醒”。
4. 当前页面没有展示相似客户列表，也没有给出“继续提交 / 去查看已有客户”的分流动作。

#### 4.3.3 规则缺口

1. 关键字去重到底基于客户名称、简称、行业词，还是多字段组合，目前未定。
2. 相似度规则未定：完全匹配、包含匹配、分词匹配、简称映射都尚未明确。
3. 提示时机未定：输入中、失焦后、保存前还是提交时。
4. 当前提示是强阻断，后续是否允许“提醒后继续提交”未定。
5. 相似客户提示至少展示哪些辅助字段未定，例如地区、客户类型、联系人、创建人。

#### 4.3.4 当前结论

该项不是完全从零开始，已有同名校验雏形，因此可以直接进入规则细化。

但进入编码前，必须先确认：

1. 匹配字段。
2. 相似规则。
3. 提示时机。
4. 阻断还是提示放行。
5. 相似客户展示字段。

## 5. 首轮规则验收结论

### 5.1 可以继续推进到设计细化的条目

1. 周报 / 周报导出。
2. 客户关键字去重提醒。

这两项都已经有真实代码锚点，不需要再讨论“系统里有没有相关能力”，下一步应直接把规则补全到可开发级别。

### 5.2 不能直接进开发的条目

1. 工作台标签收拢到工作台下方。

原因不是技术上做不到，而是业务对象尚未被唯一指认。此时直接开发，极容易做成错误对象迁移。

## 6. 建议的下一动作

建议继续按以下顺序推进：

1. 先补周报规则确认单，明确视角、导出格式、字段口径和权限边界。
2. 再补客户去重规则确认单，明确匹配字段、相似规则、提示时机和放行策略。
3. 单独拉出工作台标签对象确认单，先确认“标签”指向，再进入界面改造设计。

补充：

1. 上述三份确认文档已单独补齐，可直接作为下一轮规则验收承载件。

## 7. 统一结论

第二梯队的首轮规则验收已经有了实证基线：

1. 周报不是空白需求，而是已有入口和接口，但保存 / 导出闭环未完成。
2. 客户去重不是空白需求，而是已有“同名校验”雏形，但离“关键字去重提醒”还有一层业务规则缺口。
3. 工作台标签收拢当前最大问题不是实现，而是目标对象未唯一化。

这意味着第二梯队现在可以继续推进，但不能三项并行盲做，而应按“周报规则细化 -> 客户去重规则细化 -> 工作台标签对象确认”的顺序收口。
