# 第四阶段详细执行清单

日期：2026-03-29

阶段名称：第四阶段 测试体系、发布体系与运维硬化

目标：将第三阶段已经完成领域收敛的系统，进一步建设为可稳定验证、可稳定发布、可监控、可回滚、可恢复的生产候选版本。

## 1. 治理来源

第四阶段执行以以下文档为准：

1. `docs/plans/2026-03-29-production-hardening-and-business-refactor-plan.md`
2. `docs/plans/2026-03-29-remaining-governance-roadmap-to-pilot-and-go-live.md`
3. `docs/plans/2026-03-29-phase-3-refactor-slice-plan.md`
4. `docs/plans/phase-1-release-checklist.md`
5. `docs/plans/phase-1-rollback-checklist.md`

## 2. 阶段边界

### 本阶段要做

1. 建立发布候选版本的自动化回归基线。
2. 扩展 CI 为真实发布门禁，而不仅是最小通过线。
3. 建立日志、监控、告警和问题定位基础设施。
4. 建立环境标准化、部署标准化和运行手册。
5. 建立备份、恢复、回滚和发布演练机制。

### 本阶段不做

1. 不重新定义业务术语和状态机。
2. 不把新增业务功能混入第四阶段。
3. 不在本阶段再发散审批或方案模型设计。
4. 不在没有第三阶段主链路收敛的情况下提前做形式化运维包装。

## 3. 进入条件

开始第四阶段前，至少满足：

1. 第二阶段已完成退出评审。
2. 第三阶段核心切片已完成，并通过对应回归测试。
3. 核心主流程已有统一真相源，不再依赖旧 route 直改字段。

## 4. 退出标准

阶段结束时，至少满足以下结果：

1. 核心业务回归套件可自动执行。
2. CI 具备发布门禁能力。
3. 有可操作的发布、回滚、备份、恢复文档和演练记录。
4. 有核心监控指标、告警规则和故障排查入口。
5. 环境配置和部署流程可复现。

## 6. 任务拆分

### Task 1：定义测试金字塔与发布回归集合

目标：让测试从“局部可跑”升级为“发布前可信”。

执行要求：

1. 明确单元测试覆盖的 policy、service、adapter 边界。
2. 明确 API 测试覆盖的核心主流程与负路径。
3. 明确 E2E 覆盖的登录、项目主链路、审批、方案评审、仪表盘基础验证。
4. 明确发布冒烟套件与完整回归套件的差异。

验收：有测试分层文档和最低覆盖范围，不再临时决定测什么。

### Task 2：升级 CI 为发布门禁

目标：让 CI 成为候选版本筛选器，而不是仅做语法体检。

执行要求：

1. 固化 `typecheck`、`build`、核心单元测试、核心 API 测试、核心 E2E 冒烟为必过项。
2. 区分 PR 级校验、候选发布校验、正式发布前校验。
3. 明确失败阻断策略和负责人处理时限。

验收：形成可执行的发布门禁流水线定义。

### Task 3：建立日志、监控与告警基础面

目标：让系统进入“出问题可发现、可定位、可追踪”状态。

执行要求：

1. 定义错误监控入口和严重级别。
2. 定义接口耗时、错误率、登录失败率、审批失败率等核心观测指标。
3. 定义业务审计日志最小集合：登录、审批、阶段迁移、密码重置、关键导出。
4. 定义告警触发条件、通知对象和升级路径。

验收：有监控面板清单和告警规则清单。

### Task 4：建立环境标准化与部署标准化

目标：让运行方式从“依赖当前机器经验”升级为“新环境可复现”。

执行要求：

1. 梳理实际必需环境变量，清理历史示例与无效占位。
2. 明确开发、测试、试运行、生产环境的差异项。
3. 明确推荐部署模型，避免文档里有示例但仓库中无真实产物的状态。
4. 如采用容器化，提交真实可用的构建与运行配置；如暂不容器化，也必须提交等效的标准部署方案。

验收：新环境按文档可独立部署并完成最小冒烟。

### Task 5：升级发布、回滚、备份、恢复运行手册

目标：让“清单”变成“可以按步骤执行的运行手册”。

执行要求：

1. 在 Phase 1 发布与回滚清单基础上扩展到试运行/生产版本。
2. 明确发布窗口、责任人、观察期、回滚触发阈值。
3. 明确数据库备份、配置备份、文件备份的实际步骤。
4. 明确恢复演练方法和结果记录模板。

验收：至少完成一次备份恢复演练并留档。

### Task 6：形成试运行候选版本

目标：输出可用于第五阶段试运行的候选版本，而不是继续在主干上边改边试。

执行要求：

1. 冻结试运行前的候选范围。
2. 完成候选版本验证报告。
3. 明确第五阶段仅允许处理试运行暴露问题和阻断缺陷。

验收：形成可提交给业务侧试运行的候选版本包和验证结论。

## 7. 风险提醒

1. 如果第三阶段主链路还未收敛，第四阶段会沦为对混乱系统做包装。
2. 如果没有真实恢复演练，备份策略默认视为不可信。
3. 如果没有稳定告警规则，试运行阶段将只能被动等用户报错。

## 8. 第四阶段完成判定

第四阶段完成，意味着系统具备进入第五阶段试运行的工程条件，但不等于已经允许正式投产。

正式投产仍须经过：

1. 真实业务样本试运行
2. 指标与权限对账
3. 正式投产评审门禁
