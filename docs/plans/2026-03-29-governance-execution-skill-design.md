# 售前管理系统治理执行 SKILL 设计稿

**目标：** 为当前售前管理系统建立一套独立管理的执行型 SKILL，用来约束后续所有分析、规划、实施和验证动作，降低 agent 在长周期执行中的记忆漂移、局部幻觉和执行偏航风险。

---

## 1. 设计背景

当前项目已经进入“系统治理”阶段，而不是简单编码阶段。后续工作包括：

1. 业务规则澄清
2. 状态机与审批体系定版
3. 安全止血
4. 认证统一
5. 测试体系建立
6. 受控重构与分阶段替换

在这样的任务形态下，仅依靠对话上下文存在明显风险：

1. 上下文变长后，早期结论容易被遗忘或弱化。
2. agent 可能为了尽快推进，优先服从局部代码现状，而不是已定版方案。
3. 当实施任务跨多个阶段时，容易越过边界，把阶段性任务做成“无限扩张”。
4. 即使总方向没错，单次执行也可能偏离既定治理顺序。

因此，需要把“后续执行必须遵守的规则”固化为一套 SKILL，而不是只保留在聊天中。

## 2. SKILL 定位

这套 SKILL 的定位不是“帮助 agent 写代码更快”，而是“帮助 agent 在治理型项目中不走偏”。

它应当属于：

1. 工作流型 SKILL
2. 执行约束型 SKILL
3. 文档驱动型 SKILL
4. 适用于本仓库的 workspace 级定制能力

## 3. SKILL 的核心职责

这套 SKILL 应承担以下职责：

1. 在开始任何实施前，要求 agent 先读取指定治理文档。
2. 判断当前任务属于哪个治理阶段。
3. 检查当前任务是否超出阶段边界。
4. 指导 agent 先读方案，再读代码，再改实现。
5. 发现文档与代码冲突时，要求停止并上报，而不是自行拍板。
6. 要求每次实施后给出“与计划一致性说明”。
7. 要求关键变更同步更新相关文档与验证结果。

## 4. 不负责的内容

这套 SKILL 不应该承担以下职责：

1. 不替代业务决策本身。
2. 不替代架构设计文档。
3. 不替代具体模块的实现方案。
4. 不强行覆盖用户的新指令。
5. 不替代系统 prompt 或 workspace instruction 的通用编码规范。

它的职责是“约束执行过程”，不是“替代所有治理内容”。

## 5. 触发场景设计

建议这套 SKILL 的触发场景包括：

1. 用户要求继续落实当前售前系统的整改方案。
2. 用户要求按既定计划实施某个阶段任务。
3. 用户要求对现有系统进行重构、修复、安全整改、测试补齐、架构收敛。
4. 用户要求基于现有文档继续推进，而不是从零开始自由发挥。
5. 用户要求评审某项改动是否符合治理计划。

建议在描述中显式包含以下触发词：

- 落实方案
- 按计划执行
- 阶段实施
- 治理
- 生产化改造
- 状态机
- 审批体系
- 止血计划
- 受控重构
- 迭代升级

## 6. 依赖文档设计

这套 SKILL 需要把现有规划文档作为一级输入，而不是可选参考。

一级输入文档建议固定包括：

1. `docs/plans/2026-03-29-production-hardening-and-business-refactor-plan.md`
2. `docs/plans/2026-03-29-project-state-machine-and-approval-system-design.md`
3. `docs/plans/2026-03-29-phase-1-stop-bleeding-implementation-plan.md`
4. `docs/plans/2026-03-29-rebuild-vs-upgrade-assessment.md`
5. `ISSUE_SUMMARY_v1.md`

后续如果新增“权限矩阵”“数据迁移方案”“测试体系建设计划”，应允许通过 references 扩展纳入。

## 7. 执行流程设计

建议 SKILL 中明确以下标准流程：

### 步骤 1：先识别任务类型

判断当前请求属于：

1. 文档澄清
2. 方案设计
3. 第一阶段止血实施
4. 业务模型重构
5. 安全治理
6. 测试体系建设
7. 路线评估或复核

### 步骤 2：加载最低必要文档

根据任务类型读取最少但必要的文档，避免上下文污染，同时禁止跳过核心治理文档。

### 步骤 3：判断是否在阶段边界内

如果当前请求超出已定阶段范围，必须先指出偏差，再决定是否继续。

### 步骤 4：先对齐规则，再看代码

执行顺序必须是：

1. 先看已定方案
2. 再看现有实现
3. 再做差异判断
4. 再决定修改方案

### 步骤 5：实施时遵守最小变更原则

在当前阶段内，只允许做完成当前目标所必需的最小改动，不顺手扩大范围。

### 步骤 6：实施后必须回写结果

至少回写：

1. 改动内容
2. 对应哪份计划
3. 验证结果
4. 是否存在偏差
5. 是否需要更新后续计划

## 8. 停止条件设计

SKILL 中必须强制 agent 在以下情况停止自作主张推进：

1. 发现业务规则与设计稿冲突。
2. 发现代码实现与设计文档冲突，且无法判断谁应优先。
3. 当前任务需要跨阶段执行，但文档未授权。
4. 当前改动需要数据迁移、权限变化或对外接口变化，而计划未覆盖。
5. 当前问题只能通过新增第二套临时机制解决。
6. 当前实现会破坏已定版状态机、审批体系或认证统一策略。

## 9. 质量门槛设计

SKILL 应要求每次实施至少检查以下内容：

1. 是否符合当前阶段目标
2. 是否与上位文档一致
3. 是否引入新的双轨实现
4. 是否补了必要测试或验证
5. 是否同步更新相关文档

## 10. 输出格式设计

每次使用这套 SKILL 完成任务后，建议输出包含以下结构：

1. 当前任务归属阶段
2. 读取了哪些治理文档
3. 本次改动与哪份计划对齐
4. 做了哪些实现或分析
5. 做了哪些验证
6. 残余风险和偏差点
7. 是否建议进入下一阶段

## 11. 目录与管理方式设计

为了独立管理，建议采用以下结构：

```text
.github/
  skills/
    presales-governance-execution/
      SKILL.md
      references/
        charter.md
        document-priority.md
        execution-workflow.md
        stop-conditions.md
        acceptance-checklist.md
```

说明：

1. `SKILL.md` 只保留触发描述、核心流程和 references 导航。
2. 详细规则放入 `references/`，避免主文件过长。
3. 治理文档继续放在 `docs/plans/`，不与 SKILL 混放。
4. SKILL 目录只承载“如何执行”，不承载“业务方案正文”。

## 12. 建议的 references 划分

### `charter.md`

说明这套 SKILL 的职责、适用边界和核心原则。

### `document-priority.md`

定义文档优先级，例如：

1. 已定版治理方案优先于现有代码
2. 用户新明确指令优先于旧计划
3. 安全边界优先于开发便利
4. 状态机与审批设计优先于局部旧实现

### `execution-workflow.md`

定义标准执行流程和不同类型任务的最低阅读要求。

### `stop-conditions.md`

列出停止、自报风险、请求澄清的触发条件。

### `acceptance-checklist.md`

定义每次执行后的最小检查项和输出要求。

## 13. 后续演进建议

这套 SKILL 不应一次写死，建议按以下方式演进：

1. 先上线 v1，覆盖现阶段最关键的治理约束。
2. 在后续真实实施中记录哪些地方仍然容易走偏。
3. 将重复出现的偏差转化为新的 reference 规则。
4. 当业务治理文档扩展后，及时更新依赖文档列表。

## 14. 结论

为当前项目建立“治理执行 SKILL”是合理且必要的。

它的价值不在于提升编码速度，而在于：

1. 把治理路径变成可重复执行的协议。
2. 把文档约束转化成执行约束。
3. 降低 agent 在长周期、多阶段实施中的偏航概率。
4. 为后续真正落实生产化改造提供稳定护栏。
