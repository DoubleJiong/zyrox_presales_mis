# 第一阶段回滚清单

适用范围：2026-03-29 第一阶段止血改造发布失败或发布后出现阻断问题时的回滚操作。

## 1. 回滚前提

1. 发布后出现阻断级登录、鉴权、构建产物或关键接口异常。
2. 观察窗口内无法通过小范围热修快速恢复。
3. 回滚责任人已确认当前版本不宜继续对外提供服务。

## 2. 版本回退步骤

1. 确认当前线上版本号与上一稳定版本号。
2. 停止继续发布和灰度扩散。
3. 将应用回退到上一稳定构建产物或对应提交版本。
4. 重启服务并确认实例全部切换完成。

## 3. 配置回退步骤

1. 恢复上一稳定版本对应的环境变量配置。
2. 确认 `JWT_SECRET`、应用 URL、对象存储配置与回退版本匹配。
3. 如本次发布修改了 dev/test 相关开关，确认其在回退后恢复到安全默认值。

## 4. 数据回滚注意事项

1. 第一阶段改造原则上不做大规模数据迁移，但仍需确认会话黑名单、日志和权限相关数据是否需要保留。
2. 若必须恢复数据库，优先使用发布前已确认的备份。
3. 执行数据库恢复前，先记录当前线上故障现场与时间点。
4. 数据回滚必须由数据库责任人执行并记录。

## 5. 验证步骤

1. 回滚后重新验证登录页可达。
2. 回滚后重新验证未登录访问受保护接口返回 `401`。
3. 回滚后重新验证关键业务页面可打开。
4. 回滚后重新验证监控和日志无新的阻断异常。

## 6. 通知链

1. 通知发布负责人。
2. 通知技术负责人。
3. 通知业务接口人。
4. 记录回滚开始时间、完成时间、责任人和原因。
